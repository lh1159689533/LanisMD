# BMAD 方法论核心规则

## 概述

BMAD（Brainstorm-Map-Architect-Develop）是一种 AI 驱动的敏捷开发方法论。本规则定义了在 CodeBuddy 中应用 BMAD 的核心原则、完整角色体系和工作流程。

## 完整角色体系（9 个代理角色）

### 规划阶段角色
| 角色 | 代理 | 核心职责 |
|------|------|----------|
| 🔍 分析师 | bmad-analyst | 业务分析、需求收集、竞品分析、头脑风暴 |
| 🎯 产品经理 | bmad-product-manager | 需求定义、PRD 编写、用户故事 |
| 📋 产品负责人 | bmad-product-owner | 待办管理、优先级决策、版本规划 |
| 🎨 UX 专家 | bmad-ux-expert | 用户研究、交互设计、界面设计 |

### 设计阶段角色
| 角色 | 代理 | 核心职责 |
|------|------|----------|
| 🏗️ 架构师 | bmad-architect | 系统架构、技术选型、接口设计 |

### 开发阶段角色
| 角色 | 代理 | 核心职责 |
|------|------|----------|
| 📋 Scrum Master | bmad-scrum-master | Sprint 规划、敏捷流程、团队协调 |
| 💻 开发者 | bmad-developer | 代码实现、单元测试、Bug 修复 |
| 🧪 QA 工程师 | bmad-qa | 测试策略、测试用例、质量保证 |
| 🔍 代码审查员 | bmad-code-reviewer | 代码审查、质量检查、改进建议 |

## 四阶段工作流程

### 🎯 阶段一：Brainstorm（头脑风暴）
**目标**：探索问题空间，明确目标和边界

**参与角色**：分析师、产品经理、产品负责人

**关键活动**：
- 分析核心问题
- 识别目标用户
- 探索可能方案
- 明确约束条件
- 竞品分析

**产出物**：需求概要文档、业务分析报告

**相关技能**：`bmad-brainstorm`

---

### 🗺️ 阶段二：Map（需求映射）
**目标**：将想法转化为具体需求

**参与角色**：产品经理、产品负责人、UX 专家

**关键活动**：
- 编写用户故事
- 定义验收标准
- 确定优先级（MoSCoW / RICE）
- 识别依赖关系
- 用户体验设计
- 产品待办列表管理

**产出物**：PRD、用户故事列表、用户画像、线框图

**用户故事格式**：
```
作为 [用户角色]，
我想要 [功能/行为]，
以便 [获得的价值]。

验收标准：
- Given [前置条件]
- When [触发动作]
- Then [期望结果]
```

**相关技能**：`bmad-map`、`bmad-backlog`、`bmad-ux-design`

---

### 🏗️ 阶段三：Architect（架构设计）
**目标**：设计技术方案

**参与角色**：架构师、开发者

**关键活动**：
- 选择架构风格
- 技术栈选型
- 模块划分
- 接口设计
- 数据模型设计
- 架构决策记录（ADR）

**产出物**：架构设计文档、技术选型报告、API 设计文档

**相关技能**：`bmad-architect`

---

### 💻 阶段四：Develop（开发实现）
**目标**：迭代式实现功能

**参与角色**：Scrum Master、开发者、QA 工程师、代码审查员

**关键活动**：
- Sprint 规划和管理
- 编写测试（TDD）
- 实现代码
- 代码审查
- QA 测试
- 持续优化

**产出物**：可运行代码、测试用例、测试报告、审查报告

**相关技能**：`bmad-develop`、`bmad-scrum`、`bmad-qa-test`、`bmad-code-review`

---

## 阶段转换检查点

在进入下一阶段前，确保当前阶段的产出物已完成：

| 转换 | 检查项 |
|------|--------|
| Brainstorm → Map | ✅ 目标明确 ✅ 问题已识别 ✅ 初步方案已讨论 ✅ 竞品已分析 |
| Map → Architect | ✅ 用户故事完成 ✅ 验收标准定义 ✅ 优先级已排序 ✅ UX 设计完成 |
| Architect → Develop | ✅ 技术方案确定 ✅ 架构图完成 ✅ 接口已定义 ✅ ADR 已记录 |
| Develop → Release | ✅ 代码审查通过 ✅ 测试全部通过 ✅ 文档已更新 ✅ 质量达标 |

## 核心原则

1. **文档先行**：每个阶段产出明确的文档交付物
2. **小步快跑**：将大任务分解为可验证的小任务
3. **持续验证**：每完成一步就验证一步
4. **角色分离**：不同阶段使用不同的专业视角
5. **质量内建**：每个阶段都有验收标准
6. **敏捷迭代**：通过 Sprint 持续交付价值
7. **协作沟通**：角色之间紧密协作
