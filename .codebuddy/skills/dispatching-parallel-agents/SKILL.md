# 分派并行代理 (Dispatching Parallel Agents)

## 概述

当有多个不相关的故障（不同的测试文件、不同的子系统、不同的 Bug）时，按顺序调查它们会浪费时间。每个调查都是独立的，可以并行发生。

**核心原则：** 为每个独立的问题域分发一个代理，让它们并发工作。

## 何时使用

**适用场景：**
- 3 个以上测试文件失败，且根本原因不同
- 多个子系统独立损坏
- 每个问题都可以在没有其他上下文的情况下被理解
- 调查之间没有共享状态

**不适用场景：**
- 故障相关（修复一个可能会修复其他的）
- 需要了解完整的系统状态
- 代理之间会互相干扰

## 执行模式

### 1. 识别独立域

按损坏的内容对故障进行分组：
- 文件 A 测试工具审批流
- 文件 B 测试批处理完成行为
- 文件 C 测试中止功能

确保每个域是独立的（修复工具审批不应影响中止测试）。

### 2. 创建专注的代理任务

每个代理获得：
- **特定范围**：一个测试文件或子系统
- **清晰目标**：使这些测试通过
- **约束**：不要更改其他代码
- **预期输出**：发现和修复内容的摘要

### 3. 并行分发

同时启动多个任务（例如针对不同文件的修复任务）。

### 4. 审查与集成

当代理返回时：
1. 阅读每个摘要
2. 验证修复没有冲突
3. 运行完整测试套件
4. 整合所有更改

## 代理提示词模板

好的代理提示词应具备：
1. **专注**：一个清晰的问题域
2. **自包含**：理解问题所需的所有上下文
3. **输出具体**：明确代理应返回什么

```markdown
## 任务：修复 [文件名] 中的测试失败

**目标：** 使 [文件名] 中的所有测试通过

**当前状态：**
- 测试 X 失败，错误信息：[具体错误]
- 测试 Y 失败，错误信息：[具体错误]

**相关上下文：**
- [相关代码位置]
- [最近的更改]

**约束：**
- 只修改与这些测试直接相关的代码
- 不要修改其他文件
- 保持现有的 API 契约

**预期输出：**
- 修复了什么
- 修复方法
- 测试结果
```

## 常见错误

| ❌ 错误 | ✅ 正确 |
|--------|--------|
| "修复所有测试" | "修复 user-auth.test.ts 中的测试" |
| 无上下文 | 提供错误信息和相关代码位置 |
| 无约束 | 明确不要修改其他文件 |
| 输出模糊 | 要求具体的修复摘要 |

## 真实案例

**场景：** 重构后 3 个文件中出现 6 个测试失败

**决策：** 独立域（中止逻辑、批处理完成、竞态条件互不影响）

**分发：** 为每个文件分派一个代理

**结果：**
- 代理 1：用基于事件的等待替换了超时
- 代理 2：修复了事件结构错误
- 代理 3：添加了异步工具执行完成的等待

**集成：** 所有修复独立，无冲突，全套测试通过

**节省时间：** 并行解决了 3 个问题，而非顺序解决

## 关键优势

1. **并行化**：多个调查同时进行
2. **专注**：每个代理范围窄，需要跟踪的上下文少
3. **独立性**：代理互不干扰
4. **速度**：解决 3 个问题的时间仅相当于解决 1 个问题

## 验证

代理返回后的步骤：
1. 审查每个摘要
2. 检查冲突（是否代理编辑了相同的代码？）
3. 运行完整套件
4. 抽查（代理可能会犯系统性错误）

## 与其他技能的集成

- **配合技能**：`systematic-debugging`（用于每个代理的调试流程）
- **替代技能**：`subagent-driven-development`（用于顺序任务执行）
