# 接收代码审查 (Receiving Code Review)

## 概述

在实施代码审查建议前，必须进行严格的技术验证，而不是表现性的同意或盲目执行。

**核心原则：**
- 技术优先于社交
- 验证后实施
- 先假设后提问

## 响应模式

当收到代码审查反馈时，遵循以下步骤：

1. **阅读**：完整阅读反馈，不要做出反应
2. **理解**：用自己的话重述要求（或提问）
3. **验证**：对照代码库的实际情况进行检查
4. **评估**：这对于*当前*代码库在技术上是否合理？
5. **响应**：给出技术确认或基于理由的回绝
6. **实施**：一次处理一项，并测试每一项

## 禁止的响应 vs 正确响应

### ❌ 绝对不要

- "你完全正确！"
- "好观点！" / "很棒的反馈！"
- "让我现在就实施"（在验证之前）
- 任何感谢的表达

### ✅ 取而代之

- 重述技术要求
- 提出澄清性问题
- 如果是错误的，用技术理由回绝
- 直接开始工作（行动 > 言语）

## 处理不明确的反馈

**规则：** 如果有任何项目不清楚，**停止** - 暂时不要实施任何内容。先就不清楚的项目请求澄清。

**原因：** 项目之间可能存在关联。部分理解 = 错误实施。

**示例：**
- 用户说："修复 1-6"
- 你理解了 1, 2, 3, 6，但不清楚 4, 5
- ❌ **错误：** 先实施 1, 2, 3, 6，稍后再问 4, 5
- ✅ **正确：** "我理解项目 1, 2, 3, 6。在继续之前，我需要澄清 4 和 5。"

## 针对不同来源的处理

### 来自用户
- **信任**：理解后即可实施
- **仍需提问**：如果范围不清楚
- **无表演性同意**：跳过客套，直接行动或进行技术确认

### 来自外部审查者

**实施前检查清单：**
1. 技术上对此代码库是否正确？
2. 是否会破坏现有功能？
3. 当前实现的原因是什么？
4. 是否适用于所有平台/版本？
5. 审查者是否了解完整上下文？

**应对策略：**
- **如果建议似乎错误**：用技术理由回绝
- **如果无法轻易验证**：说明情况，询问方向
- **如果与先前决定冲突**：停止并先讨论

## YAGNI 检查

如果审查者建议"正确实现"某功能：

1. `grep` 代码库查找实际使用情况
2. **如果未使用**："此端点未被调用。删除它（YAGNI）？"
3. **如果已使用**：然后正确实现

## 实施顺序

对于多项目反馈：

1. **首先**澄清任何不清楚的地方
2. 然后按以下顺序实施：
   - 阻塞性问题（破坏性、安全性）
   - 简单修复（拼写错误、导入）
   - 复杂修复（重构、逻辑）
3. 单独测试每个修复
4. 验证没有回归

## 何时回绝

当出现以下情况时回绝：
- 建议破坏现有功能
- 审查者缺乏完整上下文
- 违反 YAGNI（未使用的功能）
- 对此技术栈在技术上不正确
- 存在遗留/兼容性原因
- 与架构决策冲突

**如何回绝：**
- 使用技术推理，而不是防御性姿态
- 提出具体问题
- 引用工作的测试/代码
- 如果涉及架构问题，请用户介入

## 确认正确的反馈

当反馈**是**正确时：
- ✅ "已修复。[变更简要描述]"
- ✅ "发现得好 - [具体问题]。已在 [位置] 修复。"
- ✅ [直接修复并在代码中展示]

## 常见错误

| 错误 | 修正 |
|------|------|
| 表演性同意 | 陈述要求或直接行动 |
| 盲目实施 | 先对照代码库验证 |
| 批量处理不测试 | 一次一项，每项都测 |
| 假设审查者是对的 | 检查是否会破坏东西 |
| 避免回绝 | 技术正确性 > 舒适度 |
| 部分实施 | 首先澄清所有项目 |
