# 测试驱动开发 (TDD)

## 概述

TDD 的核心循环：**先写测试 → 看着它失败 → 编写最少代码通过测试**

**核心原则：** 如果没有看到测试失败，你就无法确定它测试的是否是正确的东西。

**铁律：** 违反规则的字面意思就是违反规则的精神。

## 何时使用

**必须使用：**
- 新功能
- Bug 修复
- 重构
- 行为变更

**例外情况（需用户同意）：**
- 一次性原型
- 生成的代码
- 配置文件

> ⚠️ 如果你在想"这次跳过 TDD 没关系吧"，**停止**。那是在为懒惰找借口。

## 铁律

```
没有失败的测试，就没有生产代码。
```

- 如果在测试之前写了代码？**删掉它，重新开始。**
- **没有例外：**
  - 不要把它作为"参考"保留
  - 不要在写测试时"适应"它
  - 不要看它
  - **删掉意味着彻底删除**

## RED-GREEN-REFACTOR 循环

### RED - 编写失败的测试

编写一个展示预期行为的最小测试：
- 仅测试一种行为
- 名称清晰
- 使用真实代码（除非不可避免，否则不要使用 Mock）

```typescript
test('重试失败的操作 3 次', async () => {
  let attempts = 0;
  const operation = () => {
    attempts++;
    if (attempts < 3) throw new Error('fail');
    return 'success';
  };
  const result = await retryOperation(operation);
  expect(result).toBe('success');
  expect(attempts).toBe(3);
});
```

### 验证 RED - 看着它失败

**强制执行。永远不要跳过。**

运行测试并确认：
- 测试失败（不是报错）
- 失败信息符合预期
- 失败是因为缺少功能（而不是拼写错误）

- **如果测试通过？** 你是在测试现有行为。修改测试。
- **如果测试报错？** 修复错误，直到它正确失败。

### GREEN - 最小化代码

编写最简单的代码以通过测试：
- **不要**添加功能
- **不要**重构其他代码
- **不要**超越测试范围去"改进"代码

### 验证 GREEN - 看着它通过

**强制执行。**

运行测试并确认：
- 测试通过
- 其他测试仍然通过
- 输出干净（无错误或警告）

### REFACTOR - 清理

仅在 GREEN 阶段之后：
- 删除重复代码
- 改进命名
- 提取辅助方法
- **保持测试为绿色。不要添加新行为。**

### 重复

为下一个功能编写下一个失败的测试。

## 常见合理化借口

| 借口 | 现实 |
|------|------|
| "太简单了，不用测" | 简单的代码也会坏。写个测试只要 30 秒 |
| "我稍后再测" | 立即通过的测试证明不了任何东西 |
| "我已经手动测过了" | 临时 ≠ 系统。没有记录，无法复现 |
| "删掉太浪费" | 沉没成本谬误。保留未验证的代码是债务 |
| "保留作为参考" | 你会不自觉地适应它。删掉就是删掉 |
| "TDD 会拖慢我" | TDD 比 Debug 快 |

## 红色警报 - 停止并重新开始

如果出现以下情况，**删除代码，用 TDD 重新开始：**

- 在测试之前写了代码
- 在实现之后写测试
- 测试立即通过
- 无法解释测试为何失败
- "就这一次"的心态
- "这是关于精神，不是仪式"
- "TDD 太教条"

## 验证清单

工作完成前，检查以下所有项：

- [ ] 每个新函数/方法都有测试
- [ ] 在实现之前观察了每个测试的失败
- [ ] 每个测试都因预期原因而失败（功能缺失，而非拼写错误）
- [ ] 编写了最小代码来通过每个测试
- [ ] 所有测试通过
- [ ] 输出干净（无错误、警告）
- [ ] 测试使用真实代码（仅在不可避免时使用 Mock）
- [ ] 覆盖了边缘情况和错误

**如果不能勾选所有框？你跳过了 TDD。重新开始。**

## 最终规则

```
生产代码 → 必须存在测试且测试先失败
否则 → 不是 TDD
```
