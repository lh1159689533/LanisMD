# 深度追问 (Deep Asking)

## 概述

通过逐步追问来充分理解用户的真实需求和目标，在给出方案之前确保 95% 的信心水平。

**核心原则：在完全理解需求之前，严禁给出方案或开始实施。**

- 直接给方案容易偏离用户真实意图
- 模糊需求导致的返工成本远高于追问成本

## 何时使用

**强制使用场景：**
- 用户使用以下关键词触发：
  - `深度追问`、`深度问答`、`深度澄清`
  - `有疑问请反问我`、`请向我追问`
  - `帮我梳理需求`、`先别动手问清楚`
  - `需求不明确先问我`
  - `追问模式`、`反问模式`
- 用户的需求描述模糊或存在多种理解方式
- 涉及重大架构决策或不可逆变更
- 需求跨越多个模块或功能域

**不需要使用：**
- 简单明确的 bug 修复（使用 systematic-debugging）
- 用户已经给出了详细、完整的需求说明
- 纯信息查询类问题

## 流程

### 阶段 1：需求扫描

**收到用户需求后，先进行内部分析：**

1. **识别模糊点**：找出描述中不明确的地方
2. **发现歧义**：标记可能有多种理解方式的内容
3. **检查缺失信息**：列出做出决策所需但未提供的关键信息
4. **评估信心水平**：判断当前对需求的理解程度（0-100%）

### 阶段 2：逐步追问

**核心规则：一次只问一个问题，等待回答后再决定下一步。**

按优先级从以下维度追问：

1. **目标**：用户想要达成什么最终效果？
   - "你希望最终实现什么样的效果？"
   - "这个功能要解决什么问题？"

2. **范围**：涉及哪些模块/文件/功能？
   - "这个变更需要影响哪些部分？"
   - "有没有需要保持不变的区域？"

3. **约束**：有什么限制条件或偏好？
   - "有没有性能/兼容性/风格方面的要求？"
   - "是否需要兼容现有的某些行为？"

4. **上下文**：当前状态是什么？已经尝试过什么？
   - "目前的实现是怎样的？"
   - "之前尝试过哪些方案？遇到了什么问题？"

5. **验收标准**：怎么算"做完了"？
   - "你会如何验证这个功能是否正确？"
   - "有没有具体的测试场景或边界情况？"

**追问技巧：**
- 尽可能使用选择题，降低用户回答负担
- 当某个话题需要深入时，拆分为多个子问题
- 根据用户回答动态调整后续问题的方向
- 每次追问后重新评估信心水平

### 阶段 3：确认理解

**当信心水平达到 95% 时：**

1. **明确告知**："我已经充分理解了你的需求"
2. **给出需求总结**：用简洁的语言总结理解到的需求
3. **列出关键决策点**：说明在追问过程中确认的重要决策
4. **等待确认**：请用户确认总结是否准确

### 阶段 4：方案输出

**用户确认理解正确后：**

1. 给出完整、具体的方案
2. 方案应直接回应追问中确认的所有需求点
3. 如有多种可行方案，说明各自的优缺点和推荐理由

## 关键原则

| 原则 | 说明 |
|------|------|
| 一次一个问题 | 不要用多个问题让人不知所措 |
| 首选选择题 | 选择题比开放式问题更容易回答 |
| 动态调整 | 根据回答动态调整追问方向 |
| 持续评估 | 每次回答后重新评估信心水平 |
| 明确告知 | 信心达标后明确告知用户，再给方案 |
| 总结确认 | 给方案前先总结需求，等待确认 |

## 危险信号

如果你发现自己在做以下事情，**停止并重新开始**：

- 还没搞清楚需求就开始给方案
- 一次问了多个问题
- 问了无关紧要的问题，浪费用户时间
- 用户已经明确回答了，还在重复追问同一个点
- 假设用户的意图而不是去确认
- 信心不足 95% 就停止追问

## 与其他技能的集成

- **后续技能**：`brainstorming`（当需求明确后进入创意探索）
- **后续技能**：`writing-plans`（当需求明确后直接进入计划编写）
- **后续技能**：`systematic-debugging`（当确认是 bug 类问题后）
