# 头脑风暴 (Brainstorming)

## 概述

将想法转化为完整的设计和规格，通过自然的协作对话实现。

**核心原则：** 在实施任何功能之前，必须先明确需求、探索方案、验证设计。

## 何时使用

**强制使用场景：**
- 创建新功能
- 构建组件
- 添加功能
- 修改行为
- 任何创意工作

**不需要使用：**
- 简单的 bug 修复（使用 systematic-debugging）
- 代码重构（无功能变更）
- 配置修改

## 流程

### 阶段 1：理解想法

1. **检查上下文**：查看当前项目状态（文件、文档、最近的提交）
2. **提问策略**：每次只问一个问题来细化想法
   - 尽可能使用选择题
   - 开放式问题也可以
   - 如果一个话题需要深入探讨，拆分为多个问题
3. **关注重点**：目的、约束条件和成功标准

### 阶段 2：探索方案

1. **提出 2-3 种不同的方案**及其优缺点
2. **对话式呈现**：给出你的建议和理由
3. **首先推荐**：提出推荐的选项并解释原因

### 阶段 3：展示设计

1. **展示时机**：当你认为已充分理解要构建的内容时
2. **分段展示**：将设计拆分为小段（每段 200-300 词）
3. **持续确认**：在每一段之后询问"目前为止看起来是否正确？"
4. **覆盖范围**：
   - 架构
   - 组件
   - 数据流
   - 错误处理
   - 测试
5. **回溯机制**：如果有不清楚的地方，随时准备回去澄清

## 设计完成后

### 1. 文档化
- 将验证后的设计写入文件：`docs/plans/YYYY-MM-DD--design.md`
- 将设计文档提交到 git

### 2. 实施准备
询问："准备好设置实施环境了吗？"
- 使用 `using-git-worktrees` 创建隔离的工作空间
- 使用 `writing-plans` 创建详细的实施计划

## 关键原则

| 原则 | 说明 |
|------|------|
| 一次一个问题 | 不要用多个问题让人不知所措 |
| 首选选择题 | 选择题比开放式问题更容易回答 |
| 严格遵循 YAGNI | 从所有设计中移除不必要的功能 |
| 探索替代方案 | 在确定方案前，总是提出 2-3 种方法 |
| 增量验证 | 分段展示设计，并逐段验证 |
| 保持灵活 | 当某处不合理时，回去澄清 |

## 危险信号

如果你发现自己在做以下事情，**停止并重新开始**：

- 跳过需求澄清直接编码
- 只提出一种方案
- 一次性展示完整设计
- 忽略用户的反馈
- 添加用户没有要求的功能

## 与其他技能的集成

- **后续技能**：`writing-plans` → `executing-plans` 或 `subagent-driven-development`
- **可选技能**：`using-git-worktrees`（创建隔离工作区）
