# Subagent 驱动开发 (Subagent-Driven Development)

## 概述

通过为每个任务分派新的子代理来执行计划，每个任务完成后进行两阶段审查：首先是规格合规性审查，然后是代码质量审查。

**核心原则：** 每任务新子代理 + 两阶段审查 = 高质量、快速迭代

## 何时使用

**适用场景：**
- 有实施计划
- 任务基本独立
- 想在同一会话中执行

**vs. executing-plans（并行会话）：**
- 同一会话（无上下文切换）
- 每任务新子代理（无上下文污染）
- 每个任务后两阶段审查
- 更快迭代（任务间无需人工介入）

## 执行流程

### 准备阶段
1. 读取计划文件
2. 提取所有任务的完整文本和上下文
3. 创建 TodoWrite

### 每个任务的循环

```
1. 分派实现者子代理
   ↓
2. 子代理有问题？→ 是 → 回答问题，提供上下文 → 返回步骤 1
   ↓ 否
3. 子代理实现、测试、提交、自审
   ↓
4. 分派规格审查者子代理
   ↓
5. 规格合规？→ 否 → 子代理修复规格差距 → 返回步骤 4
   ↓ 是
6. 分派代码质量审查者子代理
   ↓
7. 质量审查通过？→ 否 → 子代理修复质量问题 → 返回步骤 6
   ↓ 是
8. 标记任务完成
   ↓
9. 还有更多任务？→ 是 → 返回步骤 1
   ↓ 否
10. 分派最终代码审查者
    ↓
11. 使用 finishing-a-development-branch 技能
```

## 两阶段审查

### 阶段 1：规格合规性审查
- 代码是否满足规格的所有要求？
- 是否有遗漏的需求？
- 是否有超出规格的额外功能？

### 阶段 2：代码质量审查
- 代码是否可读/可维护？
- 测试覆盖是否充分？
- 是否有性能问题？
- 是否遵循最佳实践？

## 示例工作流

```
你: 我正在使用 Subagent 驱动开发来执行此计划。

[读取计划文件: docs/plans/feature-plan.md]
[提取所有 5 个任务]
[创建 TodoWrite]

任务 1: Hook 安装脚本
[分派实现者子代理]

实现者: "在我开始之前 - hook 应该安装在用户级别还是系统级别？"

你: "用户级别 (~/.config/superpowers/hooks/)"

实现者: "明白了。正在实现..."
[稍后]
实现者: 
- 实现了 install-hook 命令
- 添加了测试，5/5 通过
- 自审: 发现我漏了 --force 标志，已添加
- 已提交

[分派规格审查者]
规格审查者: ✅ 符合规格 - 所有需求满足，无额外功能

[分派代码质量审查者]
代码审查者: 优点: 测试覆盖好，干净。问题: 无。已批准。

[标记任务 1 完成]

任务 2: 恢复模式
[分派实现者子代理]
...
```

## 优势

**vs. 手动执行：**
- 子代理自然遵循 TDD
- 每任务新上下文（无混淆）
- 并行安全（子代理不干扰）
- 子代理可以提问（工作前后）

**vs. executing-plans：**
- 同一会话（无交接）
- 持续进度（无等待）
- 自动审查检查点

**质量保障：**
- 自审在交接前捕获问题
- 两阶段审查：规格合规 + 代码质量
- 审查循环确保修复有效
- 规格合规防止过度/不足构建

## 危险信号

**绝不：**
- 跳过审查（规格合规或代码质量）
- 带着未修复的问题继续
- 并行分派多个实现子代理（会冲突）
- 让子代理读取计划文件（提供完整文本）
- 跳过场景设置上下文
- 忽略子代理问题
- 接受"差不多"的规格合规
- 跳过审查循环
- **在规格合规 ✅ 之前开始代码质量审查**
- 在任一审查有未解决问题时移动到下一任务

**如果子代理提问：**
- 清晰完整地回答
- 如需要提供额外上下文
- 不要催促他们进入实现

**如果审查者发现问题：**
- 实现者（同一子代理）修复
- 审查者再次审查
- 重复直到批准
- 不要跳过重审

## 与其他技能的集成

**必需的工作流技能：**
- `writing-plans` - 创建此技能执行的计划
- `requesting-code-review` - 审查者子代理的代码审查模板
- `finishing-a-development-branch` - 所有任务后完成开发

**子代理应使用：**
- `test-driven-development` - 子代理为每个任务遵循 TDD

**替代工作流：**
- `executing-plans` - 用于并行会话而非同会话执行
