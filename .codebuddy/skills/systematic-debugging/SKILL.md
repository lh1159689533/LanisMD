# 系统化调试 (Systematic Debugging)

## 概述

4 阶段根本原因分析流程，在修复 Bug、测试失败或意外行为时必须遵循。

**铁律：在找到根本原因之前，严禁进行任何修复。**

- 随机修复浪费时间且会产生新 Bug
- 快速补丁只会掩盖潜在问题

## 何时使用

**适用于任何技术问题：**
- 测试失败
- 生产环境 Bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**必须使用的时刻：**
- 处于时间压力下（紧急情况容易让人倾向于猜测）
- "只要快速修复一下"看起来很明显时
- 已经尝试了多次修复但均无效
- 之前的修复没有起作用
- 没有完全理解问题所在

**不要跳过的情况：**
- 问题看起来很简单（简单的 Bug 也有根本原因）
- 正在赶时间（匆忙行事保证会导致返工）
- 管理者要求"立刻"修复（系统化方法比盲目尝试更快）

## 四个阶段

### 阶段 1：根本原因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过警告或错误
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径和错误代码

2. **一致地复现**
   - 确定是否能可靠地触发问题
   - 如果不能复现，收集更多数据，不要猜测

3. **检查最近的变更**
   - 查看什么更改可能导致此问题

4. **在多组件系统中收集证据**
   - 在提出修复之前，先添加诊断工具
   - 记录每个组件边界的数据输入和输出
   - 运行一次以收集显示问题*位置*的证据
   - 分析证据以确定失败的组件

5. **追踪数据流**
   - 当错误深埋于调用堆栈中时，向上追溯直到找到源头

### 阶段 2：模式分析

**在修复之前找到规律：**

1. **找到可用的示例**：在同一代码库中找到类似的有效代码
2. **与参考进行对比**：如果是实现某种模式，请完整阅读参考实现
3. **识别差异**：列出工作代码和损坏代码之间的所有差异
4. **理解依赖关系**：理解组件需要哪些其他组件、设置、配置或环境假设

### 阶段 3：假设与测试

**科学方法：**

1. **提出单一假设**：清楚地陈述："我认为 X 是根本原因，因为 Y"
2. **最小化测试**：做出尽可能小的更改来测试假设。一次只改变一个变量
3. **继续前验证**：
   - 是否有效？是 → 进入阶段 4
   - 否 → 提出新假设，不要在此基础上叠加更多修复
4. **当你不知道时**：承认"我不理解 X"，寻求帮助或进行更多研究

### 阶段 3 → 阶段 4 转换检查点（强制）

**在写任何修复代码之前，必须在对话中显式输出以下内容。缺少任何一项则严禁进入阶段 4。**

```
[修复前检查点]
- 根因陈述：根本原因是 ____，因为 ____
- 证据来源：这是通过 [日志/复现/测试/对比分析] 确认的，具体证据是 ____
- 信心水平：____% 
- 信心 < 99%？→ 需要的额外信息：____
- 修改范围：本次只修改 ____ 这一处
```

**检查点规则：**
- 如果"信心水平"填不到 99%，**停止**，回到阶段 1 添加诊断日志
- 如果"证据来源"只能写"代码推理"而没有运行时数据，**停止**，先加日志收集证据
- 如果"修改范围"涉及多个文件或多处逻辑，**拆分**为多次单独修复
- 此检查点的作用是把隐性的思维要求变成显性的输出要求，防止在信心不足时冲动修复

### 阶段 4：实施

**修复根本原因，而非症状：**

1. **创建失败的测试用例**：必须在修复之前创建
2. **实施单一修复**：一次只做一项更改。不要进行"顺手"的改进或捆绑重构
3. **验证修复**：确保测试通过，没有破坏其他测试，问题确实已解决
4. **如果修复无效**：
   - 停止
   - 如果尝试次数 < 3：返回阶段 1，利用新信息重新分析
   - 如果尝试次数 ≥ 3：**停止并质疑架构**

### 3 次修复失败后：质疑架构

每次修复都在不同地方揭示新问题，这是架构问题的迹象：
- 不要再尝试第 4 次修复
- 讨论架构是否合理
- 是否需要重构

## 危险信号

如果你发现自己正在这样想，**停止并遵循流程**：

- "先快速修复一下，稍后再调查"
- "只要试试改 X 看看有没有用"
- "同时添加多个更改，运行测试"
- "跳过测试，我会手动验证"
- "可能是 X，让我修一下"
- "我不完全理解但这可能行得通"
- "再多试一次修复尝试"（当已经尝试了 2 次以上时）

## 常见误区反驳

| 借口 | 现实 |
|------|------|
| 问题很简单，不需要流程 | 简单问题也有根本原因。对于简单 Bug，流程执行很快 |
| 紧急情况，没时间走流程 | 系统化调试比盲目尝试（猜测-检查）更快 |
| 先试这个，然后再调查 | 第一次修复设定了模式。从一开始就做对 |
| 确认修复有效后再写测试 | 未经测试的修复不持久。测试先行能证明问题 |
| 一次做多个修复省时间 | 无法分离出什么起作用了。会导致新 Bug |

## 信心水平准则

**铁律：修复 Bug 时，对根因的信心水平必须达到 99% 才能动手修改代码。**

### 信心不足时的处理

1. **先加日志，后修代码**
   - 如果对根因判断的信心不足（< 99%），**严禁直接修复**
   - 必须先在关键路径上添加诊断日志，通过实际运行收集证据
   - 日志应覆盖：输入数据、中间计算值、分支判断条件、最终结果
   - 让用户复现问题并提供日志输出，用数据验证假设

2. **日志设计原则**
   - 添加统一前缀便于过滤（如 `[模块名]`）
   - 记录关键变量的实际值，而非仅记录"经过了这里"
   - 在所有可能的分支路径上都加日志，避免盲区
   - 日志要能回答"实际发生了什么"而非"我认为发生了什么"

3. **日志的生命周期管理**
   - 问题修复后，**必须清除所有排查日志**，不留残余
   - 修复代码和清理日志应在同一次提交中完成
   - 清理后做一次 lint 检查确保无遗漏

### 修复失败时的回退准则

**如果修复后问题未解决，必须清除本次修复代码。**

- 不要在错误的修复上继续叠加新修复
- 错误的修复会引入无用代码，增加理解成本
- 错误的修复可能引入新的隐蔽问题
- 回退到修复前状态，带着新收集的信息重新进入阶段 1

## 经验案例：DOM 容器层级错误（outline-sync）

**问题**：WYSIWYG 编辑器滚动时，左侧大纲高亮总是定位到第一个标题，无法正确跟随。

**错误的首次修复（信心 95%，不够）**：
- 假设根因是"光标同步覆盖了滚动同步"（`update()` 中 `syncActiveByCursor` 和 `syncActiveByScroll` 竞争）
- 引入了滚动状态锁 + selection 变化检测
- 结果：问题未解决

**正确的做法：加日志排查**：
- 在 `findActiveHeadingByScroll` 中记录：滚动容器类名、容器 `top` 值、每个标题的 `relativeTop`、最终结果
- 日志揭示了**真正根因**：代码使用 `view.dom.parentElement`（class=`milkdown`）作为参考容器，但它**不是滚动容器**。实际滚动容器是上层的 `editor-content`（`overflow:auto`）。`milkdown` 容器随内容滚出视口，`containerRect.top` 变成大负值，导致 `relativeTop` 计算全部错误。

**教训**：
1. "代码逻辑分析"得出的假设可能完全错误，因为忽略了运行时 DOM 结构的实际情况
2. 95% 的信心不够，剩下的 5% 恰好就是真正的根因
3. 日志在 10 秒内就暴露了真正的问题，远比"分析代码逻辑"更可靠
4. 涉及 DOM/布局/滚动的问题，**必须**通过日志验证运行时实际值，不能仅凭代码推理

## 现实影响

- **系统化方法**：15-30 分钟修复
- **随机修复方法**：2-3 小时的盲目折腾
- **首次修复成功率**：95% vs 40%
- **引入的新 Bug**：几乎为零 vs 常见
