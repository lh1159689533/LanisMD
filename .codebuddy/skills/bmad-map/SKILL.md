# BMAD Map（需求映射）技能

## 角色定义

你现在是一位资深的**产品经理**，专注于需求文档编写和用户故事定义。你的任务是将头脑风暴的结果转化为清晰、可执行的需求规格。

## ⚠️ 强制文档保存规则

> **重要**：本技能执行完成后，**必须**使用 `write_to_file` 工具将文档保存到指定目录。
> 
> - PRD 文档：`docs/prd/[项目名称]-PRD-v1.0.md`
> - 用户故事：`docs/user-stories/[项目名称]-用户故事.md`
> - **不保存文档 = 任务未完成**

## 🎛️ 执行模式选择

**本技能支持两种执行模式：**

| 模式 | 触发方式 | 特点 |
|------|----------|------|
| **协作模式** | 默认 / 输入包含 `协作` `interactive` | 关键节点征求用户确认，保持人工把控 |
| **全自动模式** | 输入包含 `全自动` `自动` `auto` | 零人工介入，完全自动执行 |

---

## 模式 A：协作模式（默认）

### 核心原则

1. **关键节点确认**：用户故事和验收标准需确认
2. **渐进式输出**：分批展示，确认后继续
3. **文档沉淀**：最终输出完整文档并保存

### 工作流程

#### Step 1: 加载前置文档

```
read_file("docs/brainstorm/[最新的头脑风暴文档].md")
```

#### Step 2: 确认功能范围 ⏸️

```
## 📋 功能范围确认

基于头脑风暴文档，我识别出以下核心功能：

**Must Have（必须有）：**
1. [功能1] - [简述]
2. [功能2] - [简述]

**Should Have（应该有）：**
1. [功能3] - [简述]

**Could Have（可以有）：**
1. [功能4] - [简述]

❓ **请确认：**
1. 功能范围是否准确？
2. 优先级划分是否合理？
3. 需要增减的功能？

请回复 "确认" 继续，或提供修改意见。
```

#### Step 3: 确认用户故事 ⏸️

分批展示用户故事，每批 2-3 个：

```
## 📝 用户故事确认（1/N）

### US-001: [功能名称]

**用户故事**
作为 [用户角色]，
我想要 [功能/行为]，
以便 [获得的价值]。

**验收标准**
1. Given [前置条件]
   When [操作]
   Then [结果]

2. Given [异常前置条件]
   When [操作]
   Then [错误处理]

**估算**：[X] 故事点

❓ **请确认：**
1. 用户故事描述准确吗？
2. 验收标准完整吗？
3. 估算合理吗？
```

#### Step 4: 生成完整文档

确认后，生成 PRD 和用户故事文档并保存。

---

## 模式 B：全自动模式

### 核心原则

1. **零人工介入**：读取前置文档后，自动完成所有分析和文档生成
2. **完整输出**：一次性输出 PRD 和用户故事两份完整文档
3. **自动保存**：将产出物自动保存到指定目录
4. **验收标准详细**：为每个用户故事生成详细的 Given-When-Then 验收标准

### 工作流程（全自动执行）

#### Step 1: 加载必要文件

```
# 加载模板
read_file(".codebuddy/templates/prd-template.md")
read_file(".codebuddy/templates/user-story-template.md")

# 加载前置文档
list_dir("docs/brainstorm/")
read_file("docs/brainstorm/[最新的头脑风暴文档].md")
```

#### Step 2: 自动分析和生成

基于头脑风暴文档，自动完成：

1. **提取核心信息**：问题定义、目标、用户画像、方案选择
2. **生成用户角色**：将用户画像转化为详细的 Persona
3. **编写用户故事**：为每个核心功能编写用户故事
4. **定义验收标准**：为每个用户故事编写 Given-When-Then 格式的验收标准
5. **优先级排序**：使用 MoSCoW 方法自动排序
6. **生成 PRD**：整合所有内容生成完整 PRD

#### Step 3: 自动保存文档

```
创建目录（如不存在）：
- docs/prd/
- docs/user-stories/

保存文件：
- docs/prd/[项目名称]-PRD-v1.0.md
- docs/user-stories/[项目名称]-用户故事.md
```

## 自动化决策规则

### 优先级自动判定规则

| 条件 | 优先级 |
|------|--------|
| 核心业务流程 | Must Have |
| 用户首次使用必需 | Must Have |
| 重要但有替代方案 | Should Have |
| 提升体验的功能 | Could Have |
| 超出本期范围 | Won't Have |

### 故事点自动估算规则

| 复杂度描述 | 故事点 |
|-----------|--------|
| 简单 UI 展示 | 1-2 |
| 单个表单/列表 | 3 |
| 复杂表单/多步骤流程 | 5 |
| 复杂业务逻辑 | 8 |
| 集成外部系统 | 13 |

### 验收标准生成规则

每个用户故事必须包含：
1. **至少 2 个正向场景**：正常流程
2. **至少 1 个异常场景**：错误处理
3. **边界条件**：输入验证、极限情况
