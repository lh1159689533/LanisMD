# 架构设计阶段提示词库

> 🏗️ BMAD 方法论 - Architect 阶段常用提示词

---

## 1. 架构设计

### 1.1 完整架构设计

```
基于以下需求：
[粘贴 PRD 或用户故事]

请作为架构师，帮我设计系统架构：

1. 推荐的架构风格（单体/微服务/其他）及原因
2. 技术栈选型建议：
   - 前端：
   - 后端：
   - 数据库：
   - 其他中间件：
3. 系统模块划分和职责
4. 核心接口设计
5. 数据模型设计
6. 非功能性需求考虑（性能、安全、扩展性）
7. 潜在技术风险及应对策略

请提供架构图和详细说明。
```

### 1.2 架构风格选择

```
我需要为以下项目选择架构风格：

项目类型：[项目描述]
预期规模：[用户量/数据量]
团队规模：[人数]
时间约束：[时间]

请对比以下架构风格并给出建议：
1. 单体架构
2. 微服务架构
3. Serverless 架构
4. 事件驱动架构

分析维度：
- 开发复杂度
- 部署和运维
- 可扩展性
- 团队技能要求
- 成本
```

## 2. 技术选型

### 2.1 技术栈选型

```
请帮我进行技术栈选型：

项目需求：
- 项目类型：[Web/Mobile/API/...]
- 预期用户量：[数量]
- 性能要求：[描述]
- 团队技术背景：[背景]

需要选型的技术：
1. 前端框架
2. 后端语言/框架
3. 数据库
4. 缓存方案
5. 消息队列
6. 搜索引擎（如需要）
7. 文件存储
8. 部署方案

对于每个技术，请提供：
- 推荐选择
- 备选方案
- 选择理由
- 注意事项
```

### 2.2 技术对比分析

```
请帮我对比以下技术方案：

场景：[使用场景]
候选方案：
- 方案A：[技术A]
- 方案B：[技术B]
- 方案C：[技术C]

对比维度：
1. 学习曲线
2. 开发效率
3. 性能表现
4. 社区活跃度
5. 生态系统
6. 长期维护性
7. 团队匹配度

请输出对比表格和最终建议。
```

### 2.3 ADR 撰写

```
请帮我撰写一个架构决策记录（ADR）：

决策背景：[背景描述]
做出的决策：[决策内容]
考虑的替代方案：[方案列表]

ADR 格式：
1. 状态
2. 背景
3. 决策
4. 理由
5. 考虑的替代方案（每个方案的优缺点）
6. 后果（正面和负面）
7. 风险
```

## 3. 模块设计

### 3.1 模块划分

```
请帮我进行系统模块划分：

系统功能：[功能列表]

请设计：
1. 模块列表及职责描述
2. 模块间的依赖关系
3. 模块边界定义
4. 通信方式
5. 模块图

原则：
- 高内聚、低耦合
- 单一职责
- 便于独立开发和测试
```

### 3.2 分层架构设计

```
请帮我设计分层架构：

系统类型：[系统类型]

请设计各层：
1. 表现层（Presentation）
2. 应用层（Application）
3. 领域层（Domain）
4. 基础设施层（Infrastructure）

每层需要说明：
- 职责范围
- 核心组件
- 层间交互规则
- 依赖方向
```

## 4. 接口设计

### 4.1 API 设计

```
请帮我设计以下功能的 API：

功能描述：[功能描述]
相关实体：[实体列表]

设计要求：
1. 遵循 RESTful 规范
2. 统一响应格式
3. 合适的 HTTP 方法和状态码
4. 清晰的 URL 设计
5. 请求/响应示例
6. 错误处理
7. 版本控制

请输出 API 文档格式。
```

### 4.2 接口规范定义

```
请帮我定义项目的 API 规范：

1. URL 命名规范
2. HTTP 方法使用规范
3. 请求格式规范
4. 响应格式规范
5. 错误码设计
6. 认证方式
7. 分页规范
8. 版本控制规范
```

### 4.3 接口文档模板

```
请为以下 API 生成详细文档：

API 名称：[名称]
功能描述：[描述]
请求方法：[GET/POST/PUT/DELETE]
URL：[URL]

文档需要包含：
1. 接口说明
2. 请求参数（Query/Body/Path）
3. 请求示例
4. 响应字段说明
5. 响应示例（成功/失败）
6. 错误码说明
7. 注意事项
```

## 5. 数据设计

### 5.1 数据模型设计

```
请帮我设计数据模型：

业务实体：
[实体列表及描述]

请设计：
1. ER 图
2. 核心表结构（字段、类型、约束）
3. 表关系
4. 索引设计
5. 数据字典
```

### 5.2 数据库选型

```
请帮我选择合适的数据库：

数据特点：
- 数据类型：[结构化/半结构化/非结构化]
- 数据量级：[预估]
- 读写比例：[读多写少/写多读少/均衡]
- 一致性要求：[强一致/最终一致]
- 查询模式：[简单查询/复杂查询/全文搜索]

候选方案：
1. 关系型：MySQL/PostgreSQL/...
2. NoSQL：MongoDB/Redis/...
3. 时序：InfluxDB/TimescaleDB/...
4. 搜索：Elasticsearch/...

请给出选型建议和理由。
```

### 5.3 缓存策略设计

```
请帮我设计缓存策略：

系统场景：[场景描述]
热点数据：[数据列表]

需要设计：
1. 缓存层次（本地缓存/分布式缓存）
2. 缓存粒度
3. 缓存更新策略
4. 缓存失效策略
5. 缓存穿透/击穿/雪崩应对
6. 缓存预热方案
```

```
请帮我设计系统的性能方案：

性能目标：
- 响应时间：[目标]
- 吞吐量：[目标]
- 并发用户：[目标]

请设计：
1. 性能优化策略
2. 缓存方案
3. 异步处理方案
4. 数据库优化
5. CDN 策略
6. 负载均衡

```

### 6.2 安全设计

```
请帮我设计系统的安全方案：

系统类型：[类型]
安全等级：[等级]
敏感数据：[数据列表]

请设计：
1. 认证方案（JWT/OAuth/Session）

3. 数据加密（传输/存储）
4. 输入验证

6. XSS/CSRF 防护

4. 服务拆分策略

6. 弹性伸缩策略
```

### 6.4 高可用设计

```
请帮我设计系统的高可用方案：

可用性目标：[99.9%/99.99%/...]

请设计：

5. 降级策略

7. 灾难恢复
```

## 8. 架构评审

### 8.1 架构自检

```
请帮我检查以下架构设计：

[架构描述]

检查维度：
1. 是否满足功能需求？
2. 是否满足非功能需求？
3. 是否过度设计？
4. 是否有单点故障？
5. 是否便于扩展？
6. 是否便于维护？
7. 安全性是否足够？
8. 成本是否合理？
```

### 8.2 风险评估

```
请评估以下架构的技术风险：

架构描述：[描述]
技术选型：[选型]

评估内容：
1. 技术复杂度风险
2. 团队技能风险
3. 性能瓶颈风险
4. 安全风险
5. 运维风险
6. 成本风险
7. 供应商风险

并给出缓解措施。
```

## 9. 总结输出

### 9.1 架构设计文档

```
请帮我生成完整的架构设计文档，包括：

1. 架构概述
2. 架构目标和约束
3. 架构风格
4. 系统架构图
5. 技术选型
6. 模块设计
7. 接口设计
8. 数据设计
9. 非功能性设计
10. 部署方案
11. 风险和应对
```

### 9.2 交接给开发团队

```
请帮我准备交接给开发团队的材料：

1. 架构概览（一页纸）
2. 技术栈清单
3. 开发规范
4. 接口文档
5. 数据库设计
6. 环境配置说明
7. 第一个 Sprint 的技术准备工作
```

---

## 使用说明

1. 在完成 Map 阶段后使用这些提示词
2. 按照整体架构 → 模块设计 → 详细设计的顺序
3. 重要决策使用 ADR 记录
4. 完成后进行架构评审
5. 确认可以进入 Develop 阶段

---

*提示：架构设计需要根据项目实际情况灵活调整，避免过度设计。*
