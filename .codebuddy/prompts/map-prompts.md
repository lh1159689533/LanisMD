# 需求映射阶段提示词库

> 🗺️ BMAD 方法论 - Map 阶段常用提示词

---

## 1. 用户故事编写

### 1.1 用户故事生成

```
基于以下需求概要：
[粘贴需求概要]

请作为产品经理，帮我：

1. 识别主要的用户角色
2. 为每个核心功能编写用户故事，格式：
   - 作为 [角色]
   - 我想要 [功能]
   - 以便 [价值]
3. 为每个用户故事定义验收标准（Given-When-Then）
4. 使用 MoSCoW 方法对功能进行优先级排序
5. 识别功能之间的依赖关系
```

### 1.2 单个功能的用户故事

```
我需要为以下功能编写详细的用户故事：

功能描述：[功能描述]
目标用户：[用户类型]
业务背景：[背景说明]

请帮我：
1. 编写用户故事（标准格式）
2. 定义至少 3 个验收标准场景
3. 考虑正常流程、边界条件和异常情况
4. 估算故事点（1-13）
5. 识别依赖和风险
```

### 1.3 用户故事拆分

```
这个用户故事太大了，需要拆分：

原始故事：[用户故事描述]
故事点：[当前估算]

请帮我按照 INVEST 原则拆分：
- Independent（独立的）
- Negotiable（可协商的）
- Valuable（有价值的）
- Estimable（可估算的）
- Small（小的）
- Testable（可测试的）

拆分后每个故事应该在 1-5 个故事点之间。
```

## 2. 验收标准定义

### 2.1 详细验收标准

```
请为以下用户故事定义详细的验收标准：

用户故事：[用户故事]

要求：
1. 使用 Given-When-Then 格式
2. 覆盖正常流程（Happy Path）
3. 覆盖边界条件
4. 覆盖异常/错误情况
5. 考虑性能要求（如适用）
6. 考虑安全要求（如适用）
```

### 2.2 验收标准审查

```
请审查以下验收标准是否完整：

用户故事：[故事描述]

当前验收标准：
[现有验收标准]

请检查：
1. 是否覆盖了所有主要场景？
2. 是否有遗漏的边界条件？
3. 是否有遗漏的异常情况？
4. 标准是否足够具体、可测试？
5. 是否有歧义需要澄清？
```

## 3. PRD 编写

### 3.1 完整 PRD 生成

```
基于以下信息，请帮我编写完整的 PRD：

项目名称：[名称]
项目背景：[背景]
目标用户：[用户]
核心功能：[功能列表]
时间约束：[时间]

PRD 需要包含：
1. 文档信息
2. 产品概述（背景、目标、成功指标）
3. 用户分析（用户画像、使用场景）
4. 功能需求（功能列表、用户故事）
5. 非功能需求（性能、安全、兼容性）
6. 约束和假设
7. 里程碑计划
8. 风险评估
```

### 3.2 PRD 章节补充

```
我需要补充 PRD 的 [章节名称] 部分：

已有内容：
[现有内容]

请帮我：
1. 完善这个章节的内容
2. 确保内容完整、专业
3. 使用清晰的结构和格式
4. 添加必要的表格或图表
```

## 4. 优先级排序

### 4.1 MoSCoW 排序

```
请帮我用 MoSCoW 方法对以下功能进行优先级排序：

功能列表：
[功能1]
[功能2]
[功能3]
...

背景信息：
- 项目目标：[目标]
- 时间约束：[时间]
- 资源约束：[资源]

请输出：
1. Must Have（必须有）
2. Should Have（应该有）
3. Could Have（可以有）
4. Won't Have（不会有）

并说明分类理由。
```

### 4.2 RICE 评分

```
请用 RICE 模型评估以下功能的优先级：

功能列表：
1. [功能1]
2. [功能2]
3. [功能3]

评估维度：
- Reach：影响多少用户？（每季度）
- Impact：对用户的影响程度？（0.25-3）
- Confidence：对估算的信心？（50%-100%）
- Effort：需要多少人月？

请输出 RICE 评分表格和最终排序。
```

## 5. 用户画像与旅程

### 5.1 用户画像创建

```
请帮我创建目标用户的用户画像：

产品类型：[产品描述]
目标市场：[市场描述]

用户画像应包含：
1. 基本信息（姓名、年龄、职业等）
2. 背景故事
3. 目标和动机
4. 痛点和挑战
5. 行为特征
6. 技术熟练度
7. 使用场景
8. 期望和顾虑

请创建 2-3 个不同类型的用户画像。
```

### 5.2 用户旅程图

```
请为以下场景创建用户旅程图：

用户画像：[用户描述]
场景：[场景描述]
目标：[用户目标]

旅程图应包含以下阶段：
1. 触发/认知
2. 考虑/研究
3. 决策
4. 行动/使用
5. 使用后/忠诚

每个阶段需要：
- 用户行为
- 触点
- 思考/疑问
- 情绪曲线
- 痛点
- 机会点
```

## 6. 需求澄清

### 6.1 需求细化

```
我有一个需求需要细化：

原始需求：[需求描述]

请帮我：
1. 澄清模糊的部分
2. 识别隐含的需求
3. 提出需要确认的问题
4. 定义明确的边界
5. 考虑相关的异常情况
```

### 6.2 需求冲突解决

```
我发现以下需求可能存在冲突：

需求A：[需求描述]
需求B：[需求描述]

请帮我：
1. 分析冲突点
2. 评估各自的重要性
3. 提出解决方案
4. 给出权衡建议
```

## 7. 依赖分析

### 7.1 功能依赖分析

```
请分析以下功能之间的依赖关系：

功能列表：
1. [功能1]
2. [功能2]
3. [功能3]
...

请输出：
1. 依赖关系图
2. 实现顺序建议
3. 关键路径
4. 并行开发可能性
```

### 7.2 外部依赖识别

```
请帮我识别这个项目的外部依赖：

项目描述：[项目描述]
功能列表：[功能列表]

需要识别：
1. 第三方服务依赖
2. 团队间依赖
3. 数据依赖
4. 基础设施依赖
5. 依赖风险和应对措施
```

## 8. 总结输出

### 8.1 Map 阶段总结

```
请帮我总结 Map 阶段的成果：

1. 用户故事清单（按优先级排序）
2. 核心验收标准
3. 依赖关系图
4. 风险清单
5. 遗留问题
6. 进入 Architect 阶段的准备清单
```

### 8.2 交接文档

```
请帮我准备交接给架构师的文档，包括：

1. 需求概览
2. 核心用户故事
3. 关键验收标准
4. 非功能性需求
5. 约束条件
6. 需要架构师关注的点
7. 待确认的技术问题
```

---

## 使用说明

1. 在完成 Brainstorm 阶段后使用这些提示词
2. 按照用户故事 → 验收标准 → PRD 的顺序进行
3. 保存所有产出物到项目文档目录
4. 完成后使用检查清单确认是否可以进入 Architect 阶段
