# GitHub/Gitee 文档同步功能

## 1. 背景与目标

LanisMD 作为一款本地 Markdown 编辑器，用户的文档目前仅存储在本地文件系统中。为了实现文档的云端备份、跨设备访问以及与他人协作，需要支持将本地文档同步到 GitHub/Gitee 远程仓库，同时也能从远程仓库拉取文档进行编辑。

**核心目标**:
- 让用户能够将本地 Markdown 文档及关联图片增量同步到 GitHub 或 Gitee 仓库
- 让用户能够直接打开远程仓库的文件夹，加载并编辑其中的 Markdown 文档和图片
- 提供简洁的配置界面，降低使用门槛

## 2. 功能需求

### 2.1 远程仓库配置管理

- **描述**: 用户可以配置 GitHub/Gitee 的远程仓库连接信息，支持多仓库管理
- **用户故事**: 作为 LanisMD 用户，我希望能够配置我的 GitHub/Gitee 仓库信息，以便后续进行文档同步
- **优先级**: P0
- **配置项**:
  - 平台选择（GitHub / Gitee）
  - Personal Access Token（PAT）
  - 仓库拥有者（owner）
  - 仓库名称（repo）
  - 目标分支（branch，默认 main）
  - 远程目标路径（可选，默认仓库根目录）
- **存储方式**: 配置保存为本地 JSON 文件，存储路径参考现有 AI 配置方案（`dirs::data_dir()/com.lanis.md/` 目录下）
- **验收标准**:
  - [ ] 提供设置界面，用户可添加/编辑/删除远程仓库配置
  - [ ] 配置以 JSON 格式持久化存储在系统数据目录（参考 `config_file.rs` 的 `config_path()` 方式）
  - [ ] 支持配置多个远程仓库
  - [ ] 配置完成后可测试连接是否成功
  - [ ] PAT 字段在 JSON 中存储（用户自行管理安全性）

### 2.2 文档上传（本地 → 远程）

- **描述**: 将当前打开的文件夹中的 Markdown 文件及其引用的图片，增量上传到已配置的远程仓库
- **用户故事**: 作为 LanisMD 用户，我希望能够将本地编辑的文档同步到 GitHub/Gitee，以便进行云端备份和跨设备访问
- **优先级**: P0
- **功能细节**:
  - 增量上传策略：仅上传新增或修改过的文件（基于文件内容 hash 或修改时间判断）
  - 支持上传的文件类型：`.md` 文件 + 图片文件（`.png`, `.jpg`, `.jpeg`, `.gif`, `.svg`, `.webp`）
  - 保持本地目录结构映射到远程仓库路径
  - 上传时自动生成 commit message（含时间戳和变更摘要）
- **验收标准**:
  - [ ] 用户可在侧边栏或菜单触发"上传到远程"操作
  - [ ] 只上传新增/修改的文件，未变更的文件不重复上传
  - [ ] 上传过程有进度提示
  - [ ] 上传成功/失败有明确的反馈提示
  - [ ] 图片文件随 Markdown 文件一同上传，保持相对路径引用关系
  - [ ] 支持选择目标远程仓库（如配置了多个）

### 2.3 文档拉取与编辑（远程 → 本地）

- **描述**: 用户可以打开远程 GitHub/Gitee 仓库的某个文件夹，加载其中的 Markdown 文档和图片，并支持编辑后同步回远程
- **用户故事**: 作为 LanisMD 用户，我希望能够直接打开远程仓库中的文档进行编辑，编辑完成后自动同步回去
- **优先级**: P0
- **功能细节**:
  - 浏览远程仓库目录结构，选择要打开的文件夹
  - 将选中文件夹的 Markdown 文件和图片拉取到本地临时/缓存目录
  - 在编辑器中正常编辑，图片可正常预览
  - 编辑完成后支持手动触发同步回远程（增量推送修改）
  - 冲突处理：如远程文件在拉取后被他人修改，提示用户选择保留本地或远程版本
- **验收标准**:
  - [ ] 提供"打开远程文件夹"入口（侧边栏或菜单）
  - [ ] 可浏览远程仓库的目录结构并选择文件夹
  - [ ] 拉取后文档可正常编辑，图片可正常显示
  - [ ] 编辑后可将修改同步回远程仓库
  - [ ] 远程文件存在冲突时给出明确提示和处理选项
  - [ ] 拉取过程有进度提示

### 2.4 同步状态指示

- **描述**: 在文件树或编辑器中展示文件的同步状态
- **用户故事**: 作为用户，我希望能够一目了然地看到哪些文件已同步、哪些有本地修改待上传
- **优先级**: P1
- **验收标准**:
  - [ ] 文件树中对已同步/待上传/冲突文件有不同的状态图标
  - [ ] 编辑器标题栏或状态栏展示当前文件同步状态

## 3. 非功能需求

- **性能**: 
  - 增量上传/拉取操作应在合理时间内完成（100 个文件 < 30s，网络正常情况下）
  - 文件 hash 计算不阻塞 UI 主线程
- **安全**: 
  - PAT 存储在本地 JSON 配置文件中（与 AI 配置方案一致，存储于系统数据目录）
  - 日志中禁止输出 PAT 明文
  - 网络请求强制使用 HTTPS
- **兼容性**: 
  - 支持 GitHub API v3 (REST) 和 Gitee API v5
  - 支持公有仓库和私有仓库
- **用户体验**:
  - 所有网络操作均为异步，不阻塞编辑器
  - 操作失败提供清晰的错误信息和解决建议

## 4. 约束与假设

- 认证方式仅支持 Personal Access Token（PAT），不实现 OAuth 和 SSH Key
- 上传/拉取文件类型限定为 Markdown 文件和常见图片格式
- 文件大小限制遵循 GitHub/Gitee API 的单文件上限（GitHub: 100MB, Gitee: 100MB）
- 假设用户对 Git 概念（仓库、分支）有基本了解
- 冲突处理采用简单策略（选择保留本地或远程），不实现复杂的合并功能
- Rust 后端负责 API 调用和文件 I/O，前端负责 UI 交互

## 5. 影响范围

- **涉及模块**:
  - 前端: 设置界面（新增配置面板）、侧边栏（新增同步入口和远程文件浏览）、状态指示组件
  - Rust 后端: 新增 `commands/sync.rs` 模块（GitHub/Gitee API 交互、文件 hash 计算、增量同步逻辑）
  - 前端服务层: 新增 `services/sync-service.ts`
  - 状态管理: 新增 `stores/sync-store.ts`
- **数据库变更**: 可能需要本地存储同步状态元数据（文件 hash 记录等）
