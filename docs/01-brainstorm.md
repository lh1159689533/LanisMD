## 1. 竞品分析

### 1.1 Typora —— 标杆产品

| 维度 | 详情 |
|------|------|
| **官网** | https://typora.io |
| **许可证** | 闭源商业软件 |
| **定价** | $14.99 一次性买断（支持 3 台设备），15 天免费试用 |
| **技术栈** | Electron + CodeMirror/Ace（0.x 版本，商业版闭源） |
| **平台** | macOS / Windows / Linux |

#### 核心特点

Typora 是「所见即所得 Markdown 编辑器」品类的开创者和标杆。其核心理念是**消除 Markdown 标记的干扰**——当用户离开正在编辑的文本段时，Markdown 标记自动隐藏，只留下格式化后的内容。

- **实时渲染（Live Rendering）**：编辑时即渲染，无需分栏预览，真正做到"所见即所得"
- **极简设计**：无多余 UI 元素，聚焦写作本身
- **丰富的格式支持**：标题、粗体/斜体/删除线、引用、有序/无序列表、任务列表、代码块、表格、图片、脚注、数学公式（KaTeX）、流程图（Mermaid）、序列图
- **导入导出**：支持 PDF / HTML / Word / Epub / OpenOffice / LaTeX 等格式
- **主题系统**：支持 CSS 自定义主题，社区有大量主题可选
- **文件管理**：侧边栏文件树、打开文件夹、最近文件
- **大纲视图**：Ctrl+Shift+1 快速调出文档大纲
- **源码模式**：一键切换到 Markdown 源码编辑视图
- **图片管理**：支持自动上传到图床（iPic、uPic 等）、复制粘贴图片

#### 优势

- ✅ 编辑体验极致流畅，是该品类的"黄金标准"
- ✅ 功能恰到好处，既不过于臃肿也不过于简陋
- ✅ 跨平台一致性好
- ✅ 社区活跃，主题和方案丰富
- ✅ 对数学公式、图表等专业内容的支持完善

#### 劣势

- ❌ 闭源，无法自定义核心行为
- ❌ 收费后部分用户转向开源替代
- ❌ 不支持多标签页（每个文件独立窗口，虽有部分实现但不完善）
- ❌ 基于 Electron，安装包较大、内存占用偏高
- ❌ 插件系统缺失，扩展能力有限
- ❌ 图片管理配置对新手不够友好

#### 市场定位

**个人写作和轻量文档编辑的首选工具**。定位于"写作者的工具"而非"知识管理系统"。适合博客写作、技术文档、日常笔记等场景。

### 1.2 MarkText —— 开源替代品

| 维度 | 详情 |
|------|------|
| **官网** | https://marktext.me |
| **GitHub** | https://github.com/marktext/marktext |
| **许可证** | MIT License（开源） |
| **定价** | 免费开源 |
| **技术栈** | Electron + Vue.js 2 + CodeMirror |
| **平台** | macOS / Windows / Linux |

#### 核心特点

MarkText 定位为"简单优雅的 Markdown 编辑器"，是 Typora 最知名的开源替代品。

- **所见即所得**：实时预览编辑体验，接近 Typora
- **多标签页**：支持多文件标签页切换
- **图片处理**：支持拖拽插入、复制粘贴、相对/绝对路径
- **导出格式**：HTML / PDF
- **主题支持**：内置多款主题，支持自定义
- **段落菜单**：输入 `/` 触发快捷命令菜单
- **模式切换**：支持源码模式、实时预览模式、专注模式

#### 优势

- ✅ 完全免费开源，MIT 许可
- ✅ 支持多标签页（这是对比 Typora 的一个亮点）
- ✅ `/` 命令菜单操作便捷
- ✅ 界面简洁美观

#### 劣势

- ❌ **项目维护状态堪忧**：自 2022 年以来 commits 大幅减少，社区普遍认为已进入低维护状态（截至 2025 年底，近半年无更新），已有 fork 尝试用 electron-vite 重构
- ❌ 启动速度比 Typora 慢
- ❌ 编辑流畅度不如 Typora，长文档有卡顿
- ❌ 不支持数学公式（KaTeX）和 Mermaid 图表
- ❌ 文档宽度不能自适应，全屏时内容只占中间固定宽度
- ❌ 内部链接跳转支持不完善
- ❌ 无中文界面

#### 市场定位

**Typora 的免费开源替代品**。适合个人日常写作，但受限于维护状态和功能不足，未能在替代 Typora 方面取得决定性成功。

### 1.3 Zettlr —— 学术写作导向

| 维度 | 详情 |
|------|------|
| **官网** | https://www.zettlr.com |
| **GitHub** | https://github.com/Zettlr/Zettlr |
| **许可证** | GPL-3.0 |
| **定价** | 免费开源 |
| **技术栈** | Electron + TypeScript + CodeMirror 6 |
| **平台** | macOS / Windows / Linux |

#### 核心特点

Zettlr 是专为**学术写作和知识管理**设计的 Markdown 编辑器，目标是成为"文字处理器的有力竞争对手"。

- **引用管理集成**：直接集成 Zotero、Citavi 等参考文献管理器
- **多格式导出**：PDF（Pandoc）、DOCX、HTML、ODT、LaTeX 等
- **Zettelkasten 笔记法**：支持双向链接、标签系统、ID 标识
- **文件管理**：强大的文件树和标签管理
- **内置搜索**：支持全局搜索和正则表达式搜索
- **统计功能**：字数统计、阅读时间估算
- **Pomodoro 计时器**：内置番茄钟功能
- **主题系统**：支持 CSS 自定义主题

#### 优势

- ✅ 学术写作体验一流，引用管理是其核心竞争力
- ✅ 完全免费开源（GPL-3.0）
- ✅ 功能丰富，导出格式全面
- ✅ 支持 Zettelkasten 方法论
- ✅ 活跃开发中（截至 2025 年底仍有频繁更新）
- ✅ 代码量大（8.84K commits），社区基础扎实

#### 劣势

- ❌ **非所见即所得**：采用传统的编辑器+预览分栏模式，不是真正的 WYSIWYG
- ❌ 界面复杂度较高，不够简洁
- ❌ 学习曲线陡峭，对普通用户不友好
- ❌ 性能一般，大文档有卡顿
- ❌ 日常写作场景下过于重量级

#### 市场定位

**研究者和学术工作者的专用工具**。适合需要管理大量引用和文献的学术写作场景，如论文、研究报告、专著等。对日常写作来说"杀鸡用牛刀"。

### 1.4 Obsidian —— 知识管理 + 编辑器

| 维度 | 详情 |
|------|------|
| **官网** | https://obsidian.md |
| **许可证** | 闭源（免费个人使用，商用需购买许可证） |
| **定价** | 个人免费；Sync $4/月（年付）或 $5/月（月付）；Publish $8/月（年付）或 $10/月（月付）；商用许可证一次买断 |
| **技术栈** | Electron + TypeScript + CodeMirror 6 |
| **平台** | macOS / Windows / Linux / iOS / Android |

#### 核心特点

Obsidian 不仅仅是一个编辑器，而是一个**本地优先的知识管理系统**，被誉为"第二大脑"。

- **本地优先**：所有笔记以纯 Markdown 文件存储在本地文件系统
- **双向链接**：`[[双链]]` 语法，支持知识图谱可视化
- **强大的插件生态**：1900+ 社区插件，可扩展为数据库、任务管理、AI 助手等
- **主题生态**：大量社区主题可自定义外观
- **多视图**：分屏、标签页、画板（Canvas）等多种视图模式
- **搜索功能**：强大的全文搜索和关系搜索
- **模板系统**：通过 Templater 等插件实现高级模板

#### 优势

- ✅ 插件生态极其丰富，几乎可以实现任何想象的功能
- ✅ 本地文件存储，数据完全自主可控
- ✅ 双向链接和知识图谱是知识管理的杀手级功能
- ✅ 个人使用完全免费
- ✅ 跨平台（含移动端）
- ✅ 活跃的社区和持续的产品迭代

#### 劣势

- ❌ **编辑体验非 WYSIWYG**：Markdown 编辑器本质上是源码编辑，有实时预览但非所见即所得
- ❌ "Obsidian Flavored Markdown" 不完全兼容标准 Markdown，迁移成本高
- ❌ 高度依赖插件，基础功能有限
- ❌ 插件质量参差不齐，可能导致稳定性问题
- ❌ 插件构建的工作流一旦深度依赖，难以迁移到其他工具
- ❌ 闭源，商业许可证价格不透明

#### 市场定位

**个人知识管理系统**。适合需要管理大量笔记、构建知识体系的用户。编辑器是系统的组成部分，而非核心卖点。

### 画像 A：独立开发者 / 技术博主「阿文」

#### 基本信息

- **角色**：全栈开发者 + 技术博客写作者
- **年龄**：28 岁
- **平台**：macOS（主力） + Windows（偶尔）
- **写作频率**：每周 3-5 篇技术笔记 / 博客

#### 背景

阿文是一名有 5 年经验的全栈开发者，同时运营着一个技术博客。他日常需要撰写技术文档、README 文件、博客文章和技术笔记。他习惯用 Markdown 写作，追求"沉浸式"的写作体验。

#### 痛点

1. **Typora 收费后**不愿付费，但找不到满意的替代品
2. **MarkText 维护停滞**，bug 无人修复，不敢在长文中使用
3. **Obsidian 非所见即所得**，编辑时总看到 Markdown 标记，打断写作思路
4. **VS Code 写 Markdown** 太"程序员化"，缺乏写作氛围
5. 多平台切换时编辑器体验不一致

#### 使用场景

- 撰写技术博客文章（含代码块、数学公式）
- 编写项目 README 和 API 文档
- 整理学习笔记（含 Mermaid 架构图、流程图）
- 编辑 GitHub Issue / PR 描述

#### 对编辑器的期望

- 打开即写，零配置
- 所见即所得，专注内容本身
- 代码高亮准确美观
- 轻量快速，不占太多内存
- 支持数学公式和 Mermaid 图表
- 快捷键高效，支持自定义

## 3. 差异化探索

### 3.1 从 Typora 借鉴的核心体验

作为标杆产品，Typora 有以下核心体验必须借鉴：

1. **无缝的 WYSIWYG 体验**
   - 编辑时显示标记，离开后自动隐藏
   - 光标所在段落保持"可编辑态"，其他段落呈现"渲染态"
   - 这是 Typora 最核心的交互创新

2. **极简 UI 设计哲学**
   - 默认隐藏所有非必要 UI（菜单栏、侧边栏）
   - 聚焦内容本身，消除视觉干扰
   - 退出聚焦模式时优雅地展示 UI 元素

3. **格式切换的流畅感**
   - 行内格式（粗体/斜体/删除线/行内代码）的即时切换
   - 快捷键操作无延迟
   - 段落类型切换（标题/列表/引用/代码块）的流畅过渡

4. **源码模式的补充**
   - 一键切换到纯 Markdown 源码视图
   - 满足需要直接编辑 Markdown 源码的高级用户

5. **导入导出能力**
   - 多种格式导出（PDF/HTML/DOCX）
   - 保持样式一致性

### 3.2 从其他竞品借鉴的优点

#### 来自 MarkText

- **多标签页**：MarkText 在这方面比 Typora 做得好，应该作为核心功能
- **`/` 命令菜单**：类似 Notion 的 slash command，快速插入各种块类型
- **段落拖拽**：通过拖拽手柄重新排列段落

#### 来自 Zettlr

- **字数/字符统计**：底部状态栏实时显示字数统计，对写作用户很实用
- **Pomodoro 集成**（P2 优先级）：番茄工作法计时器，帮助写作专注

#### 来自 Obsidian

- **快捷键系统**：Obsidian 的快捷键可搜索、可自定义，体验很好
- **命令面板**（Cmd+P）：类似 VS Code 的命令面板，快速执行任何操作
- **分屏编辑**：左右分屏同时查看/编辑不同文件或同一文件的不同位置（P2）

#### 来自 VS Code

- **命令面板（Command Palette）**：VS Code 的 Cmd+Shift+P 是效率神器
- **多光标编辑**：在代码块中多光标编辑很实用
- **集成终端概念**：在编辑器内快速执行命令（可作为远期探索方向）

### 3.3 个人学习项目独有的探索空间

作为个人学习项目，MyTypora 拥有**商业产品无法承担的探索自由**。以下是一些值得尝试的方向：

#### 🔬 实验 A：AI 辅助写作

利用本地 LLM（如 Ollama + Llama）实现：
- **续写/补全**：根据上下文自动续写内容
- **格式转换**：选中文本后转换为指定格式（如将纯文本列表转为表格）
- **智能纠错**：Markdown 语法错误自动修正
- **内容摘要**：对选中内容生成摘要

> 技术路径：通过 Tauri 的 sidecar 能力集成 Ollama CLI，或调用本地 Ollama API。

#### 🔬 实验 B：文档智能分析

- **实时文档结构分析**：自动生成文档统计（字数、段落数、图片数、链接数、公式数）
- **阅读时间估算**：基于字数和内容复杂度
- **可读性评分**：类似 Hemingway App 的可读性分析
- **Markdown 最佳实践检查**：检测过长的段落、过深的标题嵌套、缺失的 alt 文本等

#### 🔬 实验 C：增强型多标签页

超越简单的标签切换：
- **标签分组**：按项目/主题对标签分组
- **标签搜索**：快速在已打开的标签中搜索
- **标签布局**：支持垂直标签栏、水平标签栏切换
- **已关闭标签恢复**：类似浏览器"最近关闭的标签"

#### 🔬 实验 D：实时协作预览

虽然 MVP 不做实时协作，但可以探索：
- **本地 Web 预览服务器**：一键启动一个本地 HTTP 服务器，让同事通过浏览器实时查看文档渲染效果
- **共享渲染页面**：适合演示场景，避免"投屏时打字"的尴尬

#### 🔬 实验 E：写作环境增强

- **打字机模式（Typewriter Mode）**：光标始终保持在屏幕中央，文本上下滚动
- **专注模式增强**：高亮当前编辑段落，暗淡其他内容
- **环境音效**：可选的打字机声、雨声等白噪音（类似 Noisli）
- **会话目标**：设定今日写作字数目标，进度条实时显示

#### 🔬 实验 F：Tauri 原生能力探索

Tauri 相比 Electron 有独特的原生能力可以探索：
- **系统级快捷键**：全局快捷键快速打开/隐藏编辑器
- **原生通知**：保存成功、导出完成等操作的系统通知
- **原生菜单栏**：macOS 原生菜单栏集成
- **原生文件拖放**：更流畅的文件拖放体验
- **Tauri Plugin 生态**：利用社区 Tauri 插件快速集成功能（如 `tauri-plugin-store` 持久化配置）

### 3.4 差异化定位建议

综合以上分析，建议 MyTypora 的核心差异化定位为：

> **「比 Typora 更开放，比 MarkText 更优秀，比 Electron 更轻量」**

三个核心差异化维度：

1. **开源 + WYSIWYG**：填补 MarkText 维护空白后，开源 WYSIWYG Markdown 编辑器品类的空缺
2. **Tauri 轻量优势**：安装包更小、启动更快、内存占用更低（相比 Electron 方案可降低 50%+ 资源占用）
3. **现代化架构**：React + TypeScript + Milkdown 的现代前端技术栈，易于维护和扩展

### 4.2 Tauri 2.x 跨平台文件系统操作

| 评估维度 | 评估结果 | 说明 |
|----------|----------|------|
| **文件系统 API** | ⭐⭐⭐⭐⭐ 优秀 | `@tauri-apps/plugin-fs` 提供完整的文件系统操作 |

- ✅ 读写文件（`readTextFile` / `writeTextFile` / `readFile` / `writeFile`）
- ✅ 目录操作（`mkdir` / `readDir` / `remove` / `rename`）

- ✅ 监听文件变化（`watch`）
- ✅ 基础目录路径获取（AppData / Documents / Desktop 等）

**对话框**（`@tauri-apps/plugin-dialog`）：
- ✅ 打开文件对话框（支持多选、过滤扩展名）
- ✅ 保存文件对话框
- ✅ 打开文件夹对话框
- ✅ 消息对话框（确认/警告/错误）

**窗口管理**（多标签页实现方案）：
- ✅ 多窗口（每个标签页一个 WebviewWindow）
- ✅ 窗口间通信（`emit` / `listen`）
- ⚠️ 多标签页建议在前端通过 React 组件实现（非多窗口），共享同一个编辑器状态

#### 可行性结论

**✅ 完全可行。** Tauri 2.x 的文件系统 API 在功能上完全可以满足 Markdown 编辑器的需求，且基于 Rust 的后端在性能和安全性上优于 Electron 的 Node.js 方案。

### 4.4 PDF 导出方案

#### 方案对比

| 方案 | 可行性 | 优点 | 缺点 |
|------|--------|------|------|
| **A. Tauri WebView 打印** | ⚠️ 有限 | 利用系统打印对话框 | 无法静默导出；质量依赖系统打印驱动；跨平台差异大 |
| **B. tauri-plugin-printer-v2** | ⚠️ 社区插件 | 支持 PDF 打印 | 第三方插件，稳定性和长期维护不确定 |
| **C. 前端 html2pdf.js** | ✅ 可行 | 纯前端方案，不依赖原生能力 | 大文档性能可能不佳；样式还原可能有偏差 |
| **D. Rust 端 wkhtmltopdf** | ⚠️ 可行 | 服务端级别的 PDF 质量 | 需要额外安装 wkhtmltopdf 二进制，增加部署复杂度 |
| **E. Rust 端 headless_chrome** | ⚠️ 可行 | Chrome 级别的渲染质量 | 需要安装 Chrome/Chromium，违背 Tauri 轻量优势 |
| **F. Rust 端 genpdf** | ⚠️ 有限 | 纯 Rust 方案，无外部依赖 | 无法渲染 CSS/HTML，需手动排版，工作量大 |
| **G. 前端 jsPDF + html2canvas** | ✅ 可行 | 纯前端，无原生依赖 | Canvas 截图方式，对复杂排版还原度一般 |

#### 推荐方案

**分阶段实施**：

1. **MVP（P1）**：**方案 A + C 组合**
   - 默认使用 `window.print()` 调起系统打印对话框（用户可选择"保存为 PDF"）
   - 同时提供 `html2pdf.js` 选项实现前端直接导出 PDF
   - 优点：实现简单，零外部依赖

2. **增强（P2）**：**方案 G 升级**
   - 使用 `html2canvas` + `jsPDF` 实现更精细的 PDF 控制
   - 支持自定义页面大小、边距、页眉页脚
   - 预览后再导出

3. **远期探索**：**Tauri 原生方案**
   - 关注 Tauri 社区 `tauri-plugin-printer` 的发展
   - 考虑通过 Tauri sidecar 集成轻量 PDF 生成工具（如 `weasyprint`）

#### 可行性结论

**✅ 可行。** PDF 导出不需要完美方案，MVP 阶段利用浏览器打印 API 即可满足基本需求。后续可逐步优化。

## 5. 风险与机遇

### 5.1 主要风险

#### 🔴 高风险

**R1：Milkdown 编辑体验达不到 Typora 水平**

- **描述**：Milkdown 基于 ProseMirror，虽然功能强大，但 WYSIWYG 的"流畅感"需要大量细节打磨。Typora 的编辑体验是多年迭代的结果，作为个人项目很难在短期内复刻
- **影响**：如果编辑体验差（卡顿、光标跳转、格式切换延迟），用户不会选择替代 Typora
- **缓解措施**：
  - 优先实现核心编辑功能，确保流畅后再添加高级功能
  - 参考 Milkdown 官方 playground 的交互体验
  - 关注 ProseMirror 性能优化最佳实践
  - 及时处理 Milkdown 的已知性能问题（如有）

**R2：Milkdown 7.x 的大版本升级风险**

- **描述**：如果 Milkdown 发布 v8 且引入 Breaking Changes，可能需要大量迁移工作
- **影响**：项目维护成本增加
- **缓解措施**：
  - 关注 Milkdown 的发布计划和 RFC
  - 在代码中做好插件接口抽象，降低直接依赖

#### 🟡 中风险

**R3：Mermaid 集成的复杂度超预期**

- **描述**：Mermaid 在 ProseMirror NodeView 中的集成可能遇到渲染时机、交互（编辑/预览切换）、性能等问题
- **影响**：P1 功能延期
- **缓解措施**：
  - MVP 阶段可先实现 Mermaid 代码块的语法高亮，P1 再做渲染
  - 参考 Typora 的实现方式：代码块获得焦点时显示源码，失焦时渲染为图

**R4：Tauri 2.x 生态不够成熟**

- **描述**：相比 Electron，Tauri 的社区更小，遇到问题时的参考资料更少
- **影响**：开发过程中可能遇到文档缺失或 Bug 无人解答的情况
- **缓解措施**：
  - Tauri 2.x 已于 2024-10 发布稳定版，社区在快速增长
  - 遇到问题可以在 GitHub Issues 和 Discord 社区求助
  - 作为学习项目，解决问题本身也是成长

**R5：个人项目的持续性**

- **描述**：作为个人学习项目，可能因为时间、兴趣变化而中断
- **影响**：项目烂尾
- **缓解措施**：
  - 拆分合理的 MVP 目标，快速达成可用的版本
  - 保持代码整洁，即使中断也能有清晰的项目状态
  - 每个阶段都有可用的交付物，而非"全部做完才能用"

#### 🟢 低风险

**R6：跨平台兼容性问题**

- **描述**：macOS / Windows / Linux 三端的差异可能导致 UI 和行为不一致
- **影响**：用户体验不一致
- **缓解措施**：Tauri 的 Rust 后端统一了大部分差异，前端使用 Tauri API 而非原生 API

### 5.2 主要机遇

#### 🌟 战略机遇

**O1：开源 WYSIWYG Markdown 编辑器的市场空白**

- **描述**：Typora 收费后，开源 WYSIWYG Markdown 编辑器领域出现了明显的市场空白。MarkText 维护停滞，Zettlr 和 Obsidian 不是真正的 WYSIWYG。这个空白就是 MyTypora 的机会
- **价值**：如果做出一个好用的产品，有机会获得 GitHub Stars 和社区关注
- **行动建议**：在 README 中明确标注"Typora 的开源替代品"，吸引搜索流量

**O2：Tauri 生态的先发优势**

- **描述**：Tauri 正在快速增长，但目前优秀的 Tauri 应用案例还不多。做一个高质量的 Tauri 编辑器，可以成为 Tauri 社区的标杆项目
- **价值**：获得 Tauri 社区的关注和推广
- **行动建议**：在 Tauri 官方 Showcase 提交项目

**O3：个人学习项目的独特价值**

- **描述**：这不是商业产品，可以大胆尝试新功能、新技术，不必受限于"必须保证稳定性"的商业压力
- **价值**：探索 Tauri + React + Milkdown 的最佳实践，为社区提供参考
- **行动建议**：将技术决策和实践记录在博客/文档中

#### 🌟 技术机遇

**O4：Tauri 的性能优势**

- **描述**：Tauri 打包后体积约为 Electron 的 1/10（~3-5MB vs ~50-100MB），运行时内存占用也显著更低
- **价值**：成为"最轻量的 WYSIWYG Markdown 编辑器"
- **行动建议**：在性能指标上做量化对比，作为营销卖点

**O5：Milkdown 插件系统的可扩展性**

- **描述**：Milkdown 的插件系统非常灵活，几乎任何功能都可以通过插件实现
- **价值**：未来可以轻松添加新功能而不影响核心架构
- **行动建议**：设计良好的插件架构，方便后续扩展

**O6：AI 集成的实验空间**

- **描述**：本地 LLM（Ollama）的成熟使得在桌面应用中集成 AI 能力变得可行
- **价值**：AI 辅助写作是 2024-2026 年的热门趋势
- **行动建议**：P2 阶段探索 Tauri sidecar + Ollama 的 AI 写作助手

        高价值

低概率 ———————————————————— 高概率

          |

          |
        低价值

### 5.4 关键建议

1. **聚焦编辑体验**：这是项目的生命线。宁可功能少一些，也要确保核心编辑操作流畅自然
2. **快速 MVP**：先用最简方案跑通"打开文件 → 编辑 → 保存"的核心流程，再逐步添加功能
3. **性能优先**：利用 Tauri 的优势，在启动速度、内存占用上做出差异化
4. **拥抱社区**：作为开源项目，积极与 Milkdown 和 Tauri 社区互动
5. **记录学习过程**：项目文档不仅记录"做了什么"，更记录"为什么这么做"
