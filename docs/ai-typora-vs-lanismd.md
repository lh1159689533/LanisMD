# LanisMD vs Typora 生态 AI 功能差异分析

> **文档日期**：2026-04-24
> **分析范围**：LanisMD 当前实现 vs Typora 生态（typora-ai-edit + typora-copilot 插件）
> **数据来源**：LanisMD 源码审计 + typora-ai-edit / typora-copilot GitHub README + 社区反馈

***

## 一、整体架构对比

### 1.1 AI 能力接入方式

| 维度      | LanisMD                                                                     | typora-ai-edit                                                     | typora-copilot                                   |
| ------- | --------------------------------------------------------------------------- | ------------------------------------------------------------------ | ------------------------------------------------ |
| 集成方式    | **原生内置** -- AI 模块是应用核心功能的一部分，从类型定义、Rust 后端到前端 UI 全链路自研                      | **第三方插件** -- 通过 Typora 的插件机制注入，依赖 Typora 的 DOM 结构                  | **第三方插件** -- 同样通过插件注入，依赖 Typora CodeMirror 编辑器实例 |
| 后端通信    | Tauri IPC（invoke + event）→ Rust 端 reqwest SSE → 前端 Event 监听                 | 前端 JS 直接调 API（Fetch/SSE）                                           | 前端 JS 直连 OpenAI / Azure API                      |
| Key 安全性 | **系统 Keychain** 存储（macOS Keychain / Windows Credential Manager），Key 不经过前端内存 | 本地配置文件明文存储或浏览器 localStorage                                        | 本地 JSON 配置文件存储                                   |
| 模型支持    | 3 家预设（智谱 GLM / DeepSeek / 硅基流动）+ 自定义 OpenAI 兼容 Provider                     | 广泛支持：OpenAI / Azure / Claude / Gemini / 通义千问 / 文心一言 / 讯飞星火 / 零一万物等 | OpenAI / Azure / Gemini 等                        |
| 流式传输    | Rust SSE 解析 → Tauri Event 推送，支持 CancellationToken 取消                        | 前端 EventSource / Fetch ReadableStream                              | Fetch streaming                                  |

### 1.2 总结

LanisMD 在**安全性**（Keychain）和**架构自主性**（全栈可控）上有显著优势，但在**模型覆盖面**上远不如 typora-ai-edit 的"全家桶"策略。

## 二、功能对比矩阵

### 2.1 核心 AI 能力

| 功能                | LanisMD                                    | typora-ai-edit    | typora-copilot                   | 差距分析                             |
| ----------------- | ------------------------------------------ | ----------------- | -------------------------------- | -------------------------------- |
| **润色/改写**         | 有 `polish` 指令，替换选区                         | 有，支持多种改写风格        | 无                                | LanisMD 有基础实现，但缺乏风格选项（如正式/口语/学术） |
| **翻译**            | 有 `translate` 指令，自动检测中英互译                  | 有，支持指定目标语言        | 无                                | LanisMD 只做中英互译，缺少多语种选择 UI        |
| **续写**            | 有 `continue` 指令，基于光标前上下文                   | 有                 | 无                                | 功能对齐                             |
| **解释**            | 有 `explain` 指令                             | 有                 | 无                                | 功能对齐                             |
| **摘要/总结**         | **缺失**                                     | 有，支持全文/选区总结       | 无                                | LanisMD 缺少此高频需求                  |
| **转图表**           | 有 `mermaid` 指令，生成后自动插入 mermaid code\_block | 无                 | 无                                | **LanisMD 独有优势**                 |
| **转公式**           | 有 `latex` 指令，生成后自动插入 math\_block           | 无                 | 无                                | **LanisMD 独有优势**                 |
| **生成大纲**          | 有 `outline` 指令                             | 无                 | 无                                | **LanisMD 独有优势**                 |
| **自由问答/对话**       | **缺失**                                     | 有，支持自由输入 prompt   | 无                                | LanisMD 缺少自由对话能力                 |
| **Ghost Text 补全** | **缺失**                                     | 无                 | **核心功能** -- 输入时自动显示灰色补全建议，Tab 采纳 | LanisMD 完全缺失此功能，这是最大差距           |
| **自定义 Prompt**    | 有，设置页可添加自定义指令，出现在 Slash Menu               | 有，支持自定义 prompt 模板 | 无                                | 功能对齐                             |

### 2.2 功能差距总结

**LanisMD 的独特优势**（Typora 生态无法提供）：

1. **Mermaid 图表生成** -- AI 生成 Mermaid 代码后自动插入 code\_block 并触发预览
2. **LaTeX 公式生成** -- AI 生成公式后自动插入 math\_block 并渲染
3. **大纲生成** -- 一键从主题生成文档骨架
4. **API Key 安全存储** -- 系统 Keychain 级别的安全保障

**LanisMD 的主要缺失**：

1. **Ghost Text 自动补全** -- typora-copilot 的核心卖点，打字过程中实时提供补全建议
2. **自由对话** -- typora-ai-edit 支持用户输入任意 prompt
3. **摘要/总结** -- 高频需求但 LanisMD 未实现
4. **多语种翻译选择** -- 当前只做中英互译

### 3.2 AI 结果展示与交互

| 维度             | LanisMD                                             | typora-ai-edit        | typora-copilot                     |
| -------------- | --------------------------------------------------- | --------------------- | ---------------------------------- |
| **流式展示**       | **Overlay 浮层** -- 占位节点悬浮在编辑区上方，实时追加 delta 文本 + 闪烁光标 | **内联流式** -- 文本直接流入文档中 | 不适用（Ghost Text 是预测而非生成）            |
| **Ghost Text** | 无                                                   | 无                     | **灰色斜体文本** -- 紧跟光标后方显示，不影响文档内容     |
| **采纳/拒绝**      | **写入/丢弃/重试** 三按钮 -- 生成完成后显示操作按钮栏                    | 自动写入（流式过程中即写入）        | **Tab 采纳** / **Esc 拒绝** / 继续输入自动消失 |
| **取消生成**       | 有 -- 生成过程中显示「取消」按钮                                  | 有                     | 不适用                                |
| **差异对比**       | **缺失** -- replace-selection 模式直接替换，无法对比原文           | 有 -- 部分模式支持 diff 对比   | 不适用                                |
| **结果编辑**       | **缺失** -- 用户只能全量接受或丢弃                               | 有 -- 可以在弹窗中编辑 AI 结果   | 不适用                                |
| **历史记录**       | 有 -- 右侧面板展示最近 50 条 AI 生成记录（sessionStorage）          | 无                     | 无                                  |

### 3.3 交互差距总结

**LanisMD 交互亮点**：

1. **Slash Menu 集成** -- 与编辑器原生命令系统融合，发现性好
2. **选区浮动工具栏 AI 子菜单** -- 选中文字后一键操作，零学习成本
3. **Alt 调参面板** -- Power User 友好，可临时微调生成参数
4. **写入/丢弃/重试三选一** -- 比自动写入更安全，不会污染文档
5. **首次使用引导** -- Key 未配置时优雅引导到设置页
6. **AI 历史面板** -- 方便回溯生成结果

**LanisMD 交互短板**：

1. **无 Ghost Text** -- 这是最影响"AI 融入写作流"的缺失
2. **无快捷键** -- 高频用户无法快速触发 AI 功能
3. **无右键菜单** -- Typora 用户的肌肉记忆入口
4. **无 Diff 对比** -- 润色场景下用户看不到修改了什么
5. **无结果编辑** -- 只能全量接受或全量丢弃

## 六、改进建议

以下是基于差异分析得出的可执行改进建议，按功能领域分组：

### 6.1 核心功能补齐

#### 建议 1：Ghost Text 自动补全

**问题**：这是 LanisMD 与 typora-copilot 最大的差距。Ghost Text 让 AI 融入写作流，无需用户主动触发。

**改进方案**：

* 实现 ProseMirror Decoration 层的 ghost text 渲染（灰色斜体，不写入文档模型）

* 用户停止输入 300-500ms 后自动请求补全

* Tab 采纳 / Esc 拒绝 / 继续输入自动消失

* 需在 Rust 端实现请求去重（debounce）和前一请求自动取消

* 配置项：开关、延迟时间、触发最少字符数

**技术要点**：

* 使用 `DecorationSet` 的 widget decoration 渲染 ghost text

* 与现有 slash-menu、tooltip-toolbar 的键盘事件不冲突

* 请求频率控制（避免 API 费用暴涨）

#### 建议 2：摘要/总结指令

**问题**：高频需求但 LanisMD 未实现。

**改进方案**：

* 添加 `summarize` 指令到 `AI_COMMANDS`

* 支持选区摘要（选中文字后总结）和全文摘要（无选区时总结整个文档）

* insertMode: `insert-after`

#### 建议 3：自由对话/自定义 Prompt 就地输入

**问题**：用户有时想用 AI 做预设指令之外的事情。

**改进方案**：

* 在 Slash Menu 中添加 "AI 自由输入" 条目

* 触发后弹出与 `arg-input` 类似的输入框，用户输入任意 prompt

* 选中文字时自动将选区作为上下文传入

### 6.2 交互增强

#### 建议 4：快捷键支持

**问题**：高频用户无法快速触发 AI 功能。

**改进方案**：

* 全局快捷键：`Cmd/Ctrl + Shift + A` 打开 AI 命令面板

* 单指令快捷键（可配置）：

  * `Cmd/Ctrl + Shift + P` -- 润色

  * `Cmd/Ctrl + Shift + T` -- 翻译

  * `Cmd/Ctrl + Shift + W` -- 续写

#### 建议 5：Diff 对比视图

**问题**：润色（replace-selection）场景下用户无法看到 AI 修改了什么。

**改进方案**：

* 在占位浮层完成后，不直接显示纯文本，而是高亮显示 diff

* 删除内容显示红色删除线，新增内容显示绿色

* 简易实现：基于字符级 diff 算法（如 `diff-match-patch`）

* 用户点击「写入」后才替换原选区

#### 建议 6：结果可编辑

**问题**：当前只能全量接受或全量丢弃 AI 结果。

**改进方案**：

* 在占位浮层中将文本区域改为 `contenteditable` 或 `textarea`

* 用户可以在写入前手动修改 AI 输出

* 保留「写入」按钮，点击后将编辑后的文本写入文档

#### 建议 7：右键菜单集成

**问题**：部分用户习惯右键触发功能。

**改进方案**：

* 在编辑器右键菜单中添加 "AI 助手" 子菜单

* 仅在有选中文字时显示需要选区的指令（润色/翻译/解释）

* 无选区时显示不需要选区的指令（续写/转图表/转公式/大纲）

### 6.3 模型与配置增强

#### 建议 8：丰富 Provider 支持

**问题**：typora-ai-edit 支持几乎所有主流模型，LanisMD 只有 3 家。

**改进方案**：

* 增加更多预设 Provider（OpenAI / Claude / Gemini / 通义千问 等）

* 已有 `custom` Provider 支持 OpenAI 兼容协议，只需丰富预设列表

* 考虑支持本地模型（Ollama / LM Studio）

#### 建议 9：多语种翻译选择

**问题**：当前翻译只做中英互译。

**改进方案**：

* 在翻译指令触发时弹出语言选择（或在 arg-input 中让用户输入目标语言）

* 预设常见语种：英语 / 日语 / 韩语 / 法语 / 德语 / 西班牙语

* 保留"自动检测"作为默认选项

#### 建议 10：Per-Command 参数配置

**问题**：不同指令适合不同的 temperature 和 maxTokens。

**改进方案**：

* 允许在设置中为每个指令单独配置参数

* 如：润色 temperature=0.3，创作 temperature=1.0

* 执行时 Alt 调参面板作为临时覆盖

### 6.4 体验打磨

#### 建议 11：Token 用量展示

**问题**：用户对 AI 使用成本无感知。

**改进方案**：

* 流式完成后在占位浮层或状态栏显示本次 token 用量

* AI 历史面板中记录每次使用的 token 数

* 后端已返回 `promptTokens` / `completionTokens`，前端只需展示

#### 建议 12：AI 历史面板增强

**问题**：当前历史面板仅展示预览文本，无法复用。

**改进方案**：

* 点击历史记录可以将结果插入到当前光标位置

* 支持复制到剪贴板

* 持久化存储（改为 localStorage 或 Tauri store，而非 sessionStorage）

#### 建议 13：Loading 状态优化

**问题**：当前生成过程中只有闪烁光标作为进度指示。

**改进方案**：

* 在状态栏显示生成进度（已生成 token 数 / 耗时）

* 占位浮层添加进度指示动画

* 考虑在长时间等待时显示"仍在生成中..."提示

## 八、typora-copilot Ghost Text 技术参考

由于 Ghost Text 是最关键的差距，以下记录 typora-copilot 的实现要点供参考：

### 核心机制

1. **触发条件**：用户停止输入约 500ms 后自动请求
2. **补全范围**：基于当前行 + 前后上下文（最多几百行）
3. **渲染方式**：在 Typora 的 CodeMirror 编辑器中通过 decoration 注入灰色文本
4. **采纳**：Tab 键 -- 将 ghost text 写入文档
5. **拒绝**：Esc / 继续输入 / 鼠标点击其他位置
6. **缓存**：相同上下文的补全结果会被缓存，避免重复请求
7. **防抖**：短时间内的多次变化只发送最后一次请求
8. **取消**：新请求自动取消前一个未完成的请求

### LanisMD 实现时的 ProseMirror 适配要点

* 使用 `Plugin` + `DecorationSet.create` 管理 ghost text decoration

* 类型：`Decoration.widget` 渲染一个灰色 `<span>` 在光标后方

* 监听 `state.selection` 和 `state.doc` 变化触发/取消补全

* 与 slash-menu 互斥：当 slash-menu 激活时不触发补全

* 与 ai-inline 互斥：当有正在进行的 AI 生成时不触发补全
