> 版本：v1.0

## 一、需求概述

在 Milkdown 编辑器中新增「附件」自定义节点，以卡片形式展示插入的文件引用。用户通过 Slash Command `/file` 选择本地文件插入到文档中，支持预览、在文件管理器中显示、删除本地文件等操作。

### 关键决策

| 决策项 | 结论 |
|--------|------|
| 功能名称 | 附件 |
| Slash Command | `/file`，显示名称"附件" |
| Markdown 序列化 | `[file:文件名](路径)` |
| 文件存储 | 可配置附件目录，未配置则存储绝对路径 |
| 节点 UI | 卡片：图标(按后缀) + 文件名 + 文件大小 + 操作按钮 |
| 操作按钮 | 预览 / 在 Finder 中显示 / 删除(删本地文件) |
| 预览方式 | 全局统一配置：系统软件打开 or 内置预览组件 |
| 内置预览 | 新 Tauri 窗口 + open-file-viewer 或类似方案 |
| 图标策略 | 按具体后缀对应图标 |
| 懒渲染 | 不需要（详见第九节分析） |

### 支持的文件后缀（~100 种，不分类，前端开源组件open-file-viewer已全部支持）

docx, docm, dotx, dotm, rtf, odt, xlsx, xlsm, ods, pptx, pptm, odp, ofd, epub, xps, mp4, webm, m3u8, flv, m2ts, mp3, wav, flac, midi, txt, md, json, jsonc, json5, ipynb, yaml, toml, ini, proto, hcl, tex, gv, http, js, ts, vue, react, css, html, py, go, rs, rb, swift, kt, dxf, dwg, step, ifc, gds, oas, oasis, gltf, glb, obj, stl, fbx, dae, 3mf, usdz, geojson, kml, kmz, gpx, shp, drawio, excalidraw, zip, rar, 7z, tar, gz, tgz, bz2, xz, eml, msg, mbox, ttf, otf, woff, woff2, psd, ai, eps, sqlite, wasm, parquet, avro, webarchive

### 注意项
1. 与链接共用[xx](xxx) 语法基础，需要注意对已有插件的影响；
2. 如果用户配置的复制文件到附件目录，则需要注意重名文件处理；
3. 通过附件节点删除时，需要提示用户会同步删除本地文件让用户确认；
4. 用户移动附件文件，附件节点无需更新，预览等操作提示用户文件不存在；
5. 注意跨平台路径兼容性；
6. 插入附件后 undo，只撤销节点，不删除本地文件；

## 三、Markdown 序列化

### 3.1 格式定义

```markdown
[file:报告.pdf](/Users/xxx/Documents/报告.pdf)
[file:数据表.xlsx](./attachments/数据表.xlsx)
```

### 3.2 与普通链接的区分

- 普通链接：`[显示文本](url)` — 文本是自由内容
- 文件块：`[file:文件名](路径)` — 固定 `file:` 前缀

### 3.3 remark 插件实现要点

**解析（markdown → ProseMirror）**：

使用自定义 remark 插件，在 mdast 树中识别 `link` 节点，检查其子节点文本是否以 `file:` 开头：
- 匹配：转换为 `file_block` mdast 节点
- 不匹配：保持为普通 link

**序列化（ProseMirror → markdown）**：

`file_block` 节点序列化为 `[file:{name}]({src})`

### 3.4 正则匹配

```typescript
// 匹配 [file:任意文件名](任意路径)
const FILE_BLOCK_REGEX = /^\[file:(.+?)\]\((.+?)\)$/;
```

## 四、ProseMirror Schema

### 4.1 NodeSpec

```typescript
// schema.ts
export const fileBlockSchema = {
  group: 'block',
  atom: true,           // 原子节点，不可内部编辑
  selectable: true,     // 可选中（支持 Delete/Backspace 删除）
  draggable: true,      // 支持拖拽排序
  attrs: {
    src: { default: '' },    // 文件路径（相对或绝对）
    name: { default: '' },   // 文件名（含后缀）
    size: { default: '' },   // 文件大小（格式化字符串，如 "2.4 MB"）
  },
  parseDOM: [{
    tag: 'div[data-type="file-block"]',
    getAttrs: (dom: HTMLElement) => ({
      src: dom.getAttribute('data-src') || '',
      name: dom.getAttribute('data-name') || '',
      size: dom.getAttribute('data-size') || '',
    }),
  }],
  toDOM: (node: Node) => ['div', {
    'data-type': 'file-block',
    'data-src': node.attrs.src,
    'data-name': node.attrs.name,
    'data-size': node.attrs.size,
  }],
};
```

### 4.2 注册方式

使用 Milkdown 的 `$node` API 定义节点 schema，配合 `$view` 注册 NodeView：

```typescript
// index.ts
import { $node, $view } from '@milkdown/kit/utils';
import { fileBlockNodeSpec } from './schema';
import { FileBlockNodeView } from './node-view';

export const fileBlockSchemaNode = $node('file_block', () => fileBlockNodeSpec);

export const fileBlockView = $view(fileBlockSchemaNode, () => {
  return (node, view, getPos) => {
    return new FileBlockNodeView(node, view, getPos);
  };
});
```

### 5.1 DOM 结构

```html
<div class="lanismd-file-block" data-type="file-block">

    <button class="lanismd-file-block__btn" title="在 Finder 中显示">📂</button>
    <button class="lanismd-file-block__btn" title="删除">🗑</button>
  </div>
</div>
```

### 5.2 核心逻辑

```typescript
// node-view.ts
class FileBlockNodeView implements NodeView {
  dom: HTMLElement;

  constructor(node: Node, view: EditorView, getPos: () => number | undefined) {
    this.dom = this.buildDOM(node);
    this.bindEvents(node, view, getPos);
  }

  // 构建卡片 DOM（同步，极轻量，< 1ms）
  private buildDOM(node: Node): HTMLElement { ... }

  // 绑定三个按钮的事件
  private bindEvents(node: Node, view: EditorView, getPos: () => number | undefined) {
    // 预览按钮：根据全局配置决定行为
    //   - system: 调用 open_file_with_system
    //   - builtin: 调用 Tauri 打开新窗口

    // 在 Finder 中显示：调用现有 reveal_in_finder 命令

    // 删除按钮：
    //   1. 确认对话框（防误删）
    //   2. 调用 Rust 删除本地文件
    //   3. 从文档中删除此节点（ProseMirror transaction）
  }

  update(node: Node): boolean {
    if (node.type.name !== 'file_block') return false;
    // 更新 DOM 中的文件名、大小等
    return true;
  }

  destroy() {
    // 清理事件监听
  }

  stopEvent(): boolean { return false; }
  ignoreMutation(): boolean { return true; }
}
```

### 5.3 异步数据加载

文件大小通过 Tauri 命令异步获取。策略：
- 插入时即获取 size 并存入节点 attrs（首选，避免每次渲染都要查询）
- 若 attrs.size 为空（旧文档兼容），NodeView 构造时异步获取并更新 attrs

```typescript
// 文件大小为空时异步加载（仅此一个异步点）
if (!node.attrs.size && node.attrs.src) {
  getFileSize(node.attrs.src).then(size => {
    const pos = getPos();
    if (pos !== undefined) {
      const tr = view.state.tr.setNodeMarkup(pos, undefined, { ...node.attrs, size });
      view.dispatch(tr);
    }
  });
}
```

### 6.2 插入逻辑 (insert-command.ts)

```typescript
async function insertFileBlock(view: EditorView) {
  removeSlashTrigger(view);

  // 1. 打开系统文件选择对话框（Tauri Dialog API）
  const filePath = await open({
    multiple: false,
    filters: [], // 不限制文件类型，允许选择任何文件
  });
  if (!filePath) return; // 用户取消

  // 2. 获取文件名
  const fileName = filePath.split('/').pop() || filePath;

  // 3. 根据配置决定存储策略
  const config = useSettingsStore.getState().config;
  let finalPath = filePath;
  if (config.attachment?.storageDir) {
    finalPath = await copyFileToAttachments(filePath, config.attachment.storageDir);
  }

  // 4. 获取文件大小
  const size = await getFileSize(filePath);

  // 5. 插入 file_block 节点
  const { state, dispatch } = view;
  const schema = state.schema;
  const fileBlockType = schema.nodes.file_block;
  if (!fileBlockType) return;

  const { $from } = state.selection;
  const parent = $from.parent;
  const fileNode = fileBlockType.create({ src: finalPath, name: fileName, size });

  if (parent.type === schema.nodes.paragraph && parent.content.size === 0) {
    const from = $from.before();
    const to = $from.after();
    const tr = state.tr.replaceWith(from, to, fileNode);
    dispatch(tr.scrollIntoView());
  } else {
    const insertPos = $from.after();
    const tr = state.tr.insert(insertPos, fileNode);
    dispatch(tr.scrollIntoView());
  }
  view.focus();
}
```

// copy_file_to_attachments

// 3. 使用 std::fs::copy 复制文件

// open_file_with_system
// macOS: std::process::Command::new("open").arg(&path)
// Windows: std::process::Command::new("cmd").args(["/C", "start", "", &path])

## 九、性能分析 — 不需要懒渲染

### 9.1 与代码块对比

| 指标 | 代码块 (CodeMirror) | 文件块 (附件) |
|------|---------------------|---------------|
| 单个 NodeView 实例化耗时 | ~75ms（创建 CodeMirror 编辑器） | < 1ms（拼接静态 DOM） |
| 典型数量 | 24 个/文档 → 累计 ~1800ms | 1~5 个/文档 → 累计 < 5ms |
| 实例化内容 | 完整代码编辑器 | div + img + span + button |
| 子编辑器 | 是（CodeMirror） | 否（纯静态） |

### 9.2 结论

文件块 NodeView 的构造仅涉及：
1. 创建 1 个 div 容器
2. 创建 1 个 SVG 图标
3. 创建 2 个 span（文件名 + 大小）
4. 创建 3 个 button + 绑定 click 事件

总耗时 < 1ms，即使 50 个文件块也不超过 50ms。**引入 IntersectionObserver 懒渲染的复杂度远超收益，不实施。**

唯一的异步点：`get_file_size_formatted` 是 Tauri IPC 调用，采用先渲染骨架后填充数据的模式即可。

## 十、内置预览窗口

### 10.1 方案

当配置为 `previewMode: 'builtin'` 时，点击预览按钮：
1. 调用 Tauri 的 `WebviewWindow` API 创建新窗口
2. 新窗口加载应用的 `/preview` 路由（需要在前端添加）
3. 通过 URL 参数传递文件路径：`/preview?file=encodeURIComponent(path)`
4. 预览页面使用 `@nicepkg/gpt-runner` 的 open-file-viewer 组件或其他方案渲染

### 10.2 窗口配置

```typescript
const previewWindow = new WebviewWindow('file-preview', {
  url: `/preview?file=${encodeURIComponent(filePath)}`,
  title: fileName,
  width: 900,
  height: 700,
  center: true,
  resizable: true,
});
```

### 10.3 预览页面技术选型

参考 `open-file-viewer` 项目：
- PDF 预览：`pdfjs-dist`
- Office 文档：通过 iframe 加载在线转换服务 or 本地解析
- 代码/文本：使用 CodeMirror 只读模式 + 语法高亮
- 图片/视频/音频：原生 HTML5 标签
- 其他格式：显示文件基本信息 + 引导用户使用系统软件打开

> 注：内置预览的具体实现可作为后续迭代，第一版优先实现「系统软件打开」模式。

## 十五、风险与注意事项

| 风险 | 应对措施 |
|------|----------|
| 文件路径包含中文/特殊字符 | Rust 侧使用 `Path` 类型处理，前端 URI 编码 |
| 附件目录不存在 | 复制前自动创建目录 |
| 文件被外部移动/删除 | NodeView 渲染时检测文件是否存在，不存在时显示"文件缺失"状态 |
| 删除操作误触 | 必须弹出二次确认对话框 |
| 内置预览组件对部分格式支持不完善 | 第一版优先使用系统软件打开，内置预览作为渐进增强 |
| markdown 中 `[file:xxx](path)` 与普通链接的歧义 | `file:` 前缀是强标识，remark 解析优先级高于普通链接 |
