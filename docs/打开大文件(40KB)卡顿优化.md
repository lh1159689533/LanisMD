# 代码块懒加载 & 编辑器实例复用优化 — 技术总结

> 日期：2026-06-26  
> 范围：Milkdown 编辑器性能优化  
> 状态：已完成

---

## 1. 问题背景

### 1.1 现象

当打开包含大量代码块（code block）或 Mermaid 图表的 Markdown 文档时，编辑器出现明显的**首屏加载卡顿**，表现为：

- 编辑器初始化耗时过长（文档越长越明显）
- 切换文件时出现白屏/闪烁
- 滚动到包含代码块的区域时有短暂卡顿

### 1.2 根因分析

1. **代码高亮模块同步加载**：所有代码块在编辑器初始化时同步创建 NodeView 并加载语法高亮（Shiki/Prism），即使代码块不在可视区域内
2. **编辑器实例频繁销毁重建**：每次切换文件都完全销毁 Milkdown 编辑器实例并重新创建，导致大量 DOM 操作和插件重新初始化
3. **Mermaid 渲染阻塞**：Mermaid 图表在 NodeView 创建时立即执行渲染，阻塞主线程

---

## 2. 优化方案

### 2.1 代码块懒加载（核心优化）

**策略**：基于 Intersection Observer 的视口检测，只有代码块进入可视区域时才触发语法高亮渲染。

**实现要点**：

- 新建 `src/editor/plugins/code-block-lazy.ts` 插件
- 使用 ProseMirror Plugin + Decoration 机制，为所有 `code_block` 节点添加占位 DOM
- 通过 `IntersectionObserver` 监听占位元素是否进入视口
- 进入视口后移除占位、触发实际的语法高亮渲染
- 提供 `rootMargin` 预加载区域（默认 200px），实现"提前加载"效果

**关键代码路径**：

```
src/editor/plugins/code-block-lazy.ts   — 懒加载插件主逻辑
src/styles/editor/code-block-lazy.css   — 占位样式（骨架屏效果）
src/editor/editor-setup.ts              — 插件注册
```

### 2.2 编辑器实例复用

**策略**：切换文件时不销毁编辑器实例，而是通过 `replaceAll` 命令替换文档内容。

**实现要点**：

- `use-editor.ts` 中维护编辑器实例引用，切换文件时调用 Milkdown 的 `replaceAll` 命令
- 仅在编辑器未初始化或配置变更时才执行完整的 `create()` 流程
- 通过 `useRef` 跟踪当前加载的文件路径，避免重复加载

**关键代码路径**：

```
src/editor/hooks/use-editor.ts          — 实例管理 & 内容替换逻辑
src/editor/components/EditorCore.tsx    — 组件层适配
```

### 2.3 Mermaid 块懒渲染

**策略**：Mermaid NodeView 同样采用 Intersection Observer 延迟渲染。

**实现要点**：

- `mermaid-block/index.ts` 中的 NodeView 创建时仅渲染占位容器
- 通过 Observer 检测进入视口后才调用 `mermaid.render()`
- 渲染完成后断开 Observer 监听，释放资源

**关键代码路径**：

```
src/editor/plugins/mermaid-block/index.ts  — Mermaid 懒渲染逻辑
```

## 4. 排查手段

### 4.1 性能定位工具

| 工具 | 用途 |
|------|------|
| Chrome DevTools → Performance | 录制编辑器加载 Timeline，定位长任务（Long Task） |
| Performance Monitor（实时面板） | 观察 DOM 节点数量变化和 JS Heap |
| React DevTools → Profiler | 定位 React 组件级别的重渲染 |

### 4.2 排查流程

```
1. Performance 录制 → 发现初始化阶段有 200ms+ 的长任务
2. 展开 Call Tree → 定位到 Shiki 语法高亮 + NodeView 批量实例化
3. 对比有/无代码块的文档 → 确认代码块数量与卡顿正相关
4. 逐步验证：
   - 注释掉代码高亮 → 加载时间减半 ✓（确认高亮是主要耗时）
   - 注释掉 Mermaid render → 长文档进一步提速 ✓
   - 文件切换时 console.log 编辑器 create/destroy → 确认频繁重建 ✓
```

### 4.3 验证方式

- **定量**：Performance 录制对比优化前后的 LCP / TBT 指标
- **定性**：手动打开 500+ 行含 20 个代码块的文档，观察首屏速度和滚动流畅度
- **回归**：验证代码块编辑、语法切换、Mermaid 实时预览等功能未受影响

## 5. 技术决策 & 取舍

### 5.1 为什么选 IntersectionObserver 而非虚拟滚动？

| 方案 | 优势 | 劣势 |
|------|------|------|
| IntersectionObserver 懒加载 | 实现简单，与 ProseMirror 兼容好，不改变 DOM 结构 | 已渲染的节点不会被回收 |
| 虚拟滚动（Virtual Scroll） | 大文档下内存更优 | 与 ProseMirror 的 DOM ↔ State 映射严重冲突，实现复杂度极高 |

**结论**：对于文档编辑器场景，懒加载 + 实例复用已能覆盖 95% 的性能问题，虚拟滚动的收益不足以抵消其复杂度。

### 5.2 为什么用 replaceAll 而非 setState？

- `replaceAll` 是 Milkdown 官方推荐的文档替换 API，内部会正确处理 Schema 解析和事务提交
- 直接操作 ProseMirror State 需要处理 Schema 兼容性，容易引入 bug
- `replaceAll` 会触发完整的插件生命周期（decorations 重算等），保证一致性

### 5.3 rootMargin 预加载距离的选择

- 设为 `200px`：在大多数设备上能覆盖 1-2 屏的预加载，用户正常滚动时不会看到加载过程
- 过大（如 1000px）会削弱懒加载效果
- 过小（如 0px）会让用户看到明显的加载闪烁

## 6. 经验总结

### 6.1 核心经验

1. **性能优化必须正面攻克根因**：不能用 CSS 隐藏、setTimeout 推迟、loading 动画等手段"绕过"。本次的根因是"同步创建大量 NodeView + 执行语法高亮"，解决方案就是"减少首屏需要执行的工作量"。

2. **ProseMirror 生态的懒加载模式**：通过 Decoration + IntersectionObserver 的组合是标准做法，既不侵入 Schema 定义，也不影响 State 管理。

3. **编辑器实例是昂贵资源**：Milkdown/ProseMirror 编辑器的创建涉及 Schema 编译、插件初始化、DOM 树构建等重量级操作，应尽量复用实例。

### 6.2 踩坑记录

| 问题 | 原因 | 解决 |
|------|------|------|
| replaceAll 后装饰（decoration）丢失 | 旧的 DecorationSet 引用了已被替换的文档节点 | 在 replaceAll 后重新触发插件的 apply 逻辑重建 DecorationSet |
| Observer 在组件卸载后仍回调 | 未在清理阶段 disconnect | 在插件 destroy 生命周期中调用 `observer.disconnect()` |
| 占位高度与实际代码块高度不一致导致跳动 | 占位高度固定，实际高度动态 | 使用 `min-height` + 渲染后移除限制，配合 CSS transition 平滑过渡 |

### 6.3 适用场景

本方案适用于：
- 任何基于 ProseMirror 的编辑器（Milkdown、Tiptap、Remirror 等）
- 文档中包含大量"渲染成本高"的自定义 NodeView（代码高亮、图表、LaTeX 等）
- 需要频繁切换文档内容的场景（多标签、文件树导航等）
